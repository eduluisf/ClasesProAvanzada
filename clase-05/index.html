Modulo 01

¿Como ejecuta tu computadora un programa?

Antes de hablar de concurrencia, entendamos lo basico

Hardware basico

CPU y RAM — lo esencial

CPU — el cerebro

El procesador (CPU) es quien ejecuta las instrucciones de tu programa. Lee una instruccion, la ejecuta, lee la siguiente, la ejecuta.

  • Ejecuta una instruccion a la vez (por nucleo)
  • Es extremadamente rapido — millones por segundo
  • Pero solo puede hacer una cosa a la vez por nucleo

Nucleos (cores)

Tu CPU tiene varios nucleos (4, 8, 16...). Cada nucleo es como un trabajador independiente. Un nucleo = una tarea a la vez. Mas nucleos = mas tareas realmente simultaneas.

RAM — la memoria de trabajo

La RAM es donde tu programa vive mientras se ejecuta. Todo lo que tu codigo crea (variables, objetos, listas) existe en la RAM.

  • Es rapida pero temporal — se borra al apagar
  • La CPU lee y escribe en la RAM constantemente
  • Cada programa tiene su propia seccion de RAM

Analogia del escritorio

La CPU es tu cerebro. La RAM es tu escritorio — donde tienes los papeles con los que estas trabajando. El disco duro es el archivero — guardas cosas ahi, pero tienes que traerlas al escritorio para trabajar con ellas.

Procesos

¿Que es un proceso?

Un proceso = un programa ejecutandose

Cuando ejecutas python main.py, el sistema operativo crea un proceso. Ese proceso tiene:

  • Su propia seccion de RAM — variables, objetos, todo
  • Su propio espacio aislado — no puede ver la RAM de otros procesos
  • Un identificador unico (PID) — el SO lo usa para gestionarlo

Cada proceso es independiente

Si abres Chrome, Spotify y tu script de Python al mismo tiempo, son 3 procesos diferentes. Cada uno tiene su propia memoria. Si uno se cuelga, los otros siguen funcionando.

Clave

Un proceso no comparte memoria con otros procesos. Esto es bueno (seguridad) pero significa que comunicar procesos entre si es costoso.

Hilos (Threads)

¿Que es un hilo?

Un hilo = una linea de ejecucion dentro de un proceso

Un proceso puede tener multiples hilos. Cada hilo ejecuta instrucciones de forma independiente, pero todos comparten la misma memoria del proceso.

  • Mas ligeros que un proceso — crear un hilo es rapido
  • Comparten RAM — pueden leer las mismas variables
  • Pero compartir memoria puede causar problemas

Analogia: la cocina

El proceso es la cocina (espacio fisico). Los hilos son los cocineros. Todos trabajan en la misma cocina, usan las mismas herramientas, pero cada uno sigue su propia receta.

Un proceso con multiples hilos

┌────────────────────────────┐
│    PROCESO: python         │
├────────────────────────────┤
│  Memoria compartida (RAM)  │
│  variables, objetos, datos │
├────────┬────────┬──────────┤
│ Hilo 1 │ Hilo 2 │ Hilo 3  │
│ main() │ tarea1 │ tarea2  │
│   ↓    │   ↓    │   ↓     │

Modulo 02

Ejecucion secuencial

Como funciona tu codigo normalmente — una cosa despues de otra

Secuencial

Tu codigo es secuencial por defecto

Una instruccion a la vez

Cuando ejecutas un script de Python, las instrucciones se ejecutan de arriba hacia abajo, una por una. La linea 2 no empieza hasta que la linea 1 termine.

  • Cada operacion bloquea la siguiente
  • Si una tarda 5 segundos, todo el programa espera 5 segundos
  • Es predecible y facil de entender

Analogia: fila del banco

Hay una sola ventanilla. Cada persona espera su turno. Aunque haya 5 personas, solo se atiende a una a la vez. Las demas esperan sin hacer nada.

secuencial.py
import time

def tarea(nombre, duracion):
    print(f"Iniciando {nombre}...")
    time.sleep(duracion)  # simula trabajo
    print(f"{nombre} terminada.")

tarea("Descarga A", 2)
tarea("Descarga B", 2)
tarea("Descarga C", 2)

# Tiempo total: 2 + 2 + 2 = 6 segundos
# Cada descarga ESPERA a que termine
# la anterior para empezar

Secuencial — Visual

¿Que esta haciendo la CPU?

Linea de tiempo — ejecucion secuencial

Cada barra es una tarea. Las grises son tiempo esperando (la CPU no hace nada util).

0s → 2s
Descarga A (esperando red...)
2s → 4s
Descarga B (esperando red...)
4s → 6s
Descarga C (esperando red...)

Total: 6 segundos. La CPU esta parada el 99% del tiempo esperando respuesta de la red.

El problema: la espera

Cuando tu programa hace una descarga, una lectura de archivo, o una consulta a una API, la CPU no esta calculando nada. Esta esperando a que la red o el disco respondan.

  • Esto se llama operacion de I/O (Input/Output)
  • La CPU queda bloqueada sin hacer nada util
  • Mientras espera la Descarga A, podria estar haciendo la B y la C

La pregunta clave

Si las 3 descargas son independientes entre si... ¿por que no hacerlas al mismo tiempo?

Eso es exactamente lo que resuelven la concurrencia y el paralelismo.

Modulo 03

Concurrencia vs Paralelismo

No son lo mismo — entender la diferencia es fundamental

Concurrencia

¿Que es la concurrencia?

Multiples tareas en progreso al mismo tiempo

La concurrencia significa que varias tareas avanzan durante el mismo periodo de tiempo, pero no necesariamente al mismo instante exacto.

  • Un solo nucleo puede hacer concurrencia
  • La CPU alterna rapidamente entre tareas
  • Parece simultaneo, pero es turnos muy rapidos

Analogia: un mesero, muchas mesas

Un solo mesero atiende 5 mesas. No atiende todas al mismo tiempo, pero alterna entre ellas: toma el pedido de la mesa 1, mientras la cocina prepara, va a la mesa 2, luego vuelve a la 1. Todas avanzan "al mismo tiempo".

Concurrencia en la linea de tiempo

Las tareas se intercalan. Mientras una espera, otra avanza.

Tarea A
run
wait
run
Tarea B
wait
run
wait
Tarea C
wait
run
wait

Un solo nucleo alterna entre las tres. Total: ~2 segundos en vez de 6.

¿Por que funciona?

Porque mientras la Tarea A espera la respuesta de la red, la CPU puede avanzar con la Tarea B. No se desperdicia el tiempo de espera.

Paralelismo

¿Que es el paralelismo?

Multiples tareas ejecutandose al mismo instante

El paralelismo significa que varias tareas se ejecutan literalmente al mismo tiempo, cada una en un nucleo diferente de la CPU.

  • Requiere multiples nucleos
  • Cada nucleo ejecuta una tarea diferente
  • Es simultaneo de verdad, no por turnos

Analogia: muchos meseros, muchas mesas

En vez de un mesero alternando entre mesas, hay un mesero por mesa. Cada uno atiende su mesa al mismo tiempo. Mas rapido, pero necesitas mas meseros (nucleos).

Paralelismo en la linea de tiempo

Las tareas ocurren al mismo tiempo real en nucleos distintos.

Nucleo 1
Tarea A
Nucleo 2
Tarea B
Nucleo 3
Tarea C

Tres nucleos, tres tareas. Total: el tiempo de la mas lenta.

¿Cuando usar paralelismo?

Para tareas que usan mucha CPU: calculos matematicos, procesamiento de imagenes, machine learning. Donde la CPU trabaja sin parar, no espera a nadie.

Regla practica

¿Tu programa espera mucho? (APIs, archivos, red) → Concurrencia (threads/asyncio). ¿Tu programa calcula mucho? (numeros, imagenes) → Paralelismo (multiprocessing).

Modulo 04

I/O bound vs CPU bound

El tipo de tarea determina la solucion

Tipos de tarea

I/O bound vs CPU bound

I/O bound — limitado por entrada/salida

El programa pasa la mayor parte del tiempo esperando respuestas externas. La CPU esta ociosa.

  • Llamadas a APIs (HTTP requests)
  • Leer/escribir archivos en disco
  • Consultas a bases de datos
  • Descargar archivos de internet
  • Esperar input del usuario

Solucion: concurrencia

threading o asyncio. Mientras una tarea espera, otra avanza. No necesitas mas nucleos.

CPU bound — limitado por procesamiento

El programa pasa la mayor parte del tiempo calculando. La CPU esta al 100% trabajando sin parar.

  • Calculos matematicos pesados
  • Procesamiento de imagenes/video
  • Entrenamiento de modelos de ML
  • Compresion de archivos
  • Criptografia

Solucion: paralelismo

multiprocessing. Distribuyes el calculo entre varios nucleos. Cada nucleo procesa una parte.

Elegir bien

¿Por que importa la diferencia?

Si usas la herramienta equivocada...

Usar threads para una tarea CPU bound no la acelera (en Python). Usar multiprocessing para I/O bound funciona pero es excesivo e ineficiente.

El GIL de Python

Python tiene un candado llamado GIL (Global Interpreter Lock). Solo permite que un hilo ejecute Python a la vez, aunque haya multiples nucleos.

  • Para I/O bound: no importa — los hilos esperan, no compiten por la CPU
  • Para CPU bound: los hilos no ayudan — necesitas procesos separados

Guia rapida

Tipo de tareaHerramienta
I/O bound (esperar)threading / asyncio
CPU bound (calcular)multiprocessing
Amboscombinar ambos

Lo que veremos en esta clase

Nos enfocaremos en I/O bound con threading y concurrent.futures. Es el caso mas comun: hacer multiples requests a APIs, descargar datos, leer archivos. Es donde vas a ver el mayor beneficio inmediato.

APIs

¿Que es una API y para que sirve?

API = Application Programming Interface

Una forma estandar para que dos programas hablen entre si. Tu codigo pregunta, un servidor responde. Normalmente por internet, usando HTTP y devolviendo datos en formato JSON.

Analogia: el mesero del restaurante

Tu no entras a la cocina. Le dices al mesero lo que quieres, el lo pide y te trae el plato. La API es el mesero: la interfaz entre tu y el servidor, con reglas claras de como pedir.

Lo bueno

No te importa como funciona por dentro el otro sistema. Solo necesitas saber que pedir y que vas a recibir.

¿Para que las usamos?

  • Obtener datos externos: clima, dolar, noticias
  • Integrar servicios: pagos, login con Google, mapas
  • Automatizar: publicar en redes, enviar emails, crear reportes
  • Conectar tu app con bases de datos remotas
  • Consumir modelos de IA (OpenAI, Anthropic)

Los 5 metodos HTTP principales

¿Que es un metodo HTTP?

Cuando tu programa habla con una API, le dice al servidor que accion quiere hacer mediante un "verbo". Cada metodo tiene un proposito claro.

Codigos de estado mas comunes

CodigoSignificado

201Created — se creo un recurso
400Bad Request — mandaste mal los datos
404Not Found — el recurso no existe
500Server Error — fallo el servidor

Metodo¿Que hace?Ejemplo

Idea clave

La URL dice sobre que (el recurso: /posts/1). El metodo dice que hacer con el (GET, POST, etc.).

Metodo GET

GET — leer datos de una API

¿Cuando usarlo?

Siempre que quieras consultar informacion sin modificar nada: traer un post, listar usuarios, buscar productos, ver el clima, obtener tweets…

Caracteristicas

  • Seguro: no cambia nada en el servidor
  • Idempotente: llamarlo 10 veces da el mismo resultado
  • Los parametros van en la URL (query string)
  • Es el metodo mas cacheable

En esta clase

Es el metodo I/O bound clasico: pedir, esperar respuesta, repetir. Perfecto para aplicar concurrencia.

ejemplos_get.py
import requests

base = "https://jsonplaceholder.typicode.com"

# 1. Traer UN post por su id
r = requests.get(f"{base}/posts/1")
print(r.status_code)   # 200
print(r.json()["title"])

# 2. Traer TODOS los posts (lista)
r = requests.get(f"{base}/posts")

# 3. Filtrar con query params (?userId=1)

# 4. Con headers (ej: autenticacion)
r = requests.get("https://api.github.com/user",
    headers={"Authorization": "token XXX"})
Metodos de escritura

POST, PUT, PATCH, DELETE — modificar datos

POST — crear un recurso nuevo

El servidor suele devolver el recurso creado con su id y estado 201 Created.

    "title": "Mi primer post",

r = requests.put(f"{base}/posts/1",

# Cambiar solo el titulo

print(r.status_code)   # 200

r = requests.delete(f"{base}/posts/1")
print(r.status_code)   # 200

Tip: enviar datos con json=

requests convierte el dict a JSON y pone el header Content-Type por ti. No tienes que serializar a mano.

Ejemplo — Secuencial

Secuencial: consultar 5 URLs una por una

secuencial_api.py
import requests
import time

urls = [
    "https://jsonplaceholder.typicode.com/posts/1",
    "https://jsonplaceholder.typicode.com/posts/2",
    "https://jsonplaceholder.typicode.com/posts/3",
    "https://jsonplaceholder.typicode.com/posts/4",
    "https://jsonplaceholder.typicode.com/posts/5",
]

def descargar(url):
    resp = requests.get(url)
    return resp.json()["title"]

inicio = time.time()

for url in urls:
    titulo = descargar(url)
    print(titulo)

fin = time.time()
print(f"Tiempo: {fin - inicio:.2f}s")

¿Que pasa aqui?

  1. Llama a la URL 1 → espera respuesta
  2. Recibe respuesta → imprime titulo
  3. Llama a la URL 2 → espera respuesta
  4. Recibe respuesta → imprime titulo
  5. ...y asi con cada una

Cada request tarda ~200ms. Cinco requests secuenciales = ~1 segundo.

El desperdicio

Mientras esperamos la respuesta de la URL 1, no hacemos nada. Podriamos estar pidiendo la URL 2, 3, 4 y 5 al mismo tiempo.

Con 100 URLs

200ms × 100 = 20 segundos esperando. Y la CPU sin hacer nada el 99% del tiempo. Esto no escala.

Ejemplo — Concurrente

Concurrente: consultar 5 URLs al mismo tiempo

concurrente_api.py
import requests
import time
from concurrent.futures import (
    ThreadPoolExecutor, as_completed
)

urls = [
    "https://jsonplaceholder.typicode.com/posts/1",
    "https://jsonplaceholder.typicode.com/posts/2",
    "https://jsonplaceholder.typicode.com/posts/3",
    "https://jsonplaceholder.typicode.com/posts/4",
    "https://jsonplaceholder.typicode.com/posts/5",
]

def descargar(url):
    resp = requests.get(url)
    return resp.json()["title"]

inicio = time.time()

with ThreadPoolExecutor(max_workers=5) as pool:
    futuros = {pool.submit(descargar, u): u
               for u in urls}
    for f in as_completed(futuros):
        print(f.result())

fin = time.time()
print(f"Tiempo: {fin - inicio:.2f}s")

¿Que cambio?

  1. ThreadPoolExecutor(5) crea 5 hilos
  2. pool.submit() lanza cada descarga en un hilo
  3. Las 5 descargas ocurren al mismo tiempo
  4. as_completed() nos da resultados conforme terminan

El resultado

Las 5 URLs se piden al mismo tiempo. Total: ~200ms (el tiempo de la mas lenta), no 1 segundo. Con 100 URLs el ahorro es enorme.

Nota importante

El orden de los resultados puede cambiar. as_completed devuelve el que termine primero, no el que se envio primero. Si necesitas orden, usa pool.map().

ThreadPoolExecutor

¿Que hace cada parte?

ThreadPoolExecutor

Es un pool (grupo) de hilos reutilizables. En vez de crear y destruir hilos manualmente, el pool los gestiona por vos.

  • max_workers=5 → maximo 5 hilos activos
  • Si envias 100 tareas, ejecuta 5 a la vez
  • Cuando un hilo termina, toma la siguiente tarea

pool.submit(funcion, argumento)

Envia una funcion al pool para ejecutarla en un hilo. Retorna un Future — un objeto que va a tener el resultado cuando la funcion termine.

as_completed(futuros) + future.result()

as_completed itera sobre los futures en orden de finalizacion (el primero que termina, primero). .result() te da el valor retornado; si la funcion lanzo una excepcion, la vuelve a lanzar.

with ... as pool:

Al salir del bloque, el pool espera a que todas las tareas terminen y cierra los hilos limpiamente. No tienes que cerrar nada manualmente.

pool.map()

Alternativa simple: pool.map()

Cuando quieres mantener el orden

pool.map() funciona como el map() normal de Python, pero ejecuta la funcion en multiples hilos. Los resultados mantienen el orden original.

con_map.py
import requests
from concurrent.futures import ThreadPoolExecutor

base = "https://jsonplaceholder.typicode.com"
urls = [f"{base}/posts/{i}" for i in range(1, 6)]

def descargar(url):
    return requests.get(url).json()["title"]

with ThreadPoolExecutor(max_workers=5) as pool:
    # map mantiene el orden de la lista original
    for titulo in pool.map(descargar, urls):
        print(titulo)

submit() vs map()

submit + as_completedmap
Orden de finalizacionOrden original
Mas control (manejo de errores por tarea)Mas simple
Puedes enviar funciones con distinta firmaMisma funcion para todos

¿Cual usar?

  • map() cuando aplicas la misma funcion a una lista y quieres orden
  • submit() cuando necesitas mas control o las tareas son diferentes

Ambas son igual de rapidas: solo cambia como recibes los resultados.

Modulo 06

Manejo de errores

Que pasa cuando algo falla en un hilo

Errores en hilos

¿Que pasa si una tarea falla?

El error no detiene los demas hilos

Si un hilo lanza una excepcion, los demas siguen ejecutandose. El error se captura cuando llamas .result() en el Future.

errores.py
from concurrent.futures import (
    ThreadPoolExecutor, as_completed
)
import requests

urls = [
    "https://jsonplaceholder.typicode.com/posts/1",
    "https://url-que-no-existe.com/nada",
    "https://jsonplaceholder.typicode.com/posts/3",
]

def descargar(url):
    resp = requests.get(url, timeout=5)
    resp.raise_for_status()
    return resp.json()["title"]

with ThreadPoolExecutor(max_workers=3) as pool:
    futuros = {pool.submit(descargar, u): u
               for u in urls}
    for f in as_completed(futuros):
        url = futuros[f]
        try:
            titulo = f.result()
            print(f"OK: {titulo}")
        except Exception as e:
            print(f"ERROR: {url} → {e}")

¿Que pasa aqui?

  1. La URL 1 y 3 funcionan bien → imprimen titulo
  2. La URL 2 falla → el try/except captura el error
  3. Los otros hilos no se ven afectados

Buenas practicas

  • Siempre usa try/except al llamar .result()
  • Usa timeout en requests para no quedarte esperando para siempre
  • Guarda los errores para revisarlos despues

Resultado

OK: sunt aut facere...
ERROR: https://url-que-no-existe...
  → ConnectionError
OK: ea molestias quasi...

t1 = time.time()
for url in urls:
    descargar(url)
sec = time.time() - t1

# --- CONCURRENTE ---
t2 = time.time()
with ThreadPoolExecutor(max_workers=10) as pool:
    resultados = list(pool.map(descargar, urls))
con = time.time() - t2

print(f"Secuencial: {sec:.2f}s")
print(f"Concurrente: {con:.2f}s")
print(f"Speedup: {sec/con:.1f}x")

¿Que aprendemos?

  • El codigo concurrente es casi identico al secuencial
  • Solo cambia el for por pool.map()
  • La mejora en tiempo es dramatica

¿max_workers?

Con 10 workers y 20 URLs: ejecuta 10 en la primera tanda, luego los otros 10. No necesitas un hilo por URL — el pool recicla hilos automaticamente.

¿Por que pool.map() aqui?

Porque la misma funcion (descargar) se aplica a toda la lista. Es el caso perfecto para map().

Ejercicio 2
Solucion

Multiples APIs en paralelo

ejercicio2.py
import requests, time
from concurrent.futures import (
    ThreadPoolExecutor, as_completed
)

endpoints = {
    "posts": "https://jsonplaceholder.typicode.com/posts?_limit=5",
    "users": "https://jsonplaceholder.typicode.com/users?_limit=5",
    "todos": "https://jsonplaceholder.typicode.com/todos?_limit=5",
}

def consultar(nombre, url):
    resp = requests.get(url, timeout=5)
    resp.raise_for_status()
    datos = resp.json()
    return nombre, len(datos)

inicio = time.time()

with ThreadPoolExecutor(max_workers=3) as pool:
    futuros = [
        pool.submit(consultar, n, u)
        for n, u in endpoints.items()
    ]
    for f in as_completed(futuros):
        try:
            nombre, total = f.result()
            print(f"{nombre}: {total} resultados")
        except Exception as e:
            print(f"Error: {e}")

print(f"Tiempo: {time.time()-inicio:.2f}s")

¿Que aprendemos?

  • submit() permite pasar multiples argumentos
  • La funcion retorna una tupla (nombre, total)
  • El try/except protege contra fallos individuales

¿Por que submit() y no map()?

Porque cada tarea recibe dos argumentos diferentes (nombre y URL). map() solo acepta un iterable. submit() es mas flexible.

Caso de uso real

Dashboard que muestra datos de varias fuentes: un endpoint de ventas, otro de usuarios, otro de metricas. Con concurrencia, la pagina carga en el tiempo del mas lento, no en la suma de todos.

Buscar usuario concurrentemente

ejercicio3.py
import requests, time
from concurrent.futures import ThreadPoolExecutor

urls = [f"https://jsonplaceholder.typicode.com/users/{i}"
        for i in range(1, 11)]

def descargar_usuario(url):
    return requests.get(url).json()

inicio = time.time()

with ThreadPoolExecutor(max_workers=10) as pool:
    usuarios = list(pool.map(descargar_usuario, urls))

buscado = "Delphine"
for u in usuarios:
    if u["username"] == buscado:
        print(f"Encontrado: {u['name']}")
        print(f"Email: {u['email']}")
        print(f"Ciudad: {u['address']['city']}")
        break

print(f"Tiempo: {time.time()-inicio:.2f}s")

Patron comun

  1. Descargar todo concurrentemente
  2. Procesar localmente despues

La descarga es lo lento (I/O). Filtrar en memoria es instantaneo. Separar estas dos fases es un patron muy util.

¿Por que no buscar en la API directamente?

Podrias usar /users?username=Delphine (un solo request). Pero el ejercicio practica concurrencia — en el mundo real muchas APIs no tienen filtros y necesitas descargar y filtrar.

Resumen

Lo que aprendimos hoy

Conceptos

  • Proceso — programa en ejecucion, memoria aislada
  • Hilo — linea de ejecucion dentro de un proceso, memoria compartida
  • Secuencial — una tarea a la vez, simple pero lento
  • Concurrencia — tareas se intercalan, aprovecha la espera
  • Paralelismo — tareas al mismo tiempo real, multiples nucleos
  • I/O bound → threads. CPU bound → multiprocessing
