Módulo 01

¿Qué es POO?

Programación Orientada a Objetos — modelar el mundo en código

POO — El problema

¿Por qué no alcanza con funciones y variables?

Programación procedural — lo que ya conocemos

  • El programa es una lista de instrucciones que se ejecutan en orden
  • Los datos (variables) y las acciones (funciones) están separados
  • Funciona bien cuando el problema es simple y pequeño
# Procedural — datos y funciones sueltos, sin relación
nombre = "Ana"
saldo  = 1000

def depositar(saldo, monto):
    return saldo + monto

saldo = depositar(saldo, 500)
# ¿Qué pasa si tenemos 100 cuentas?
# ¿Y si queremos agregar validaciones?

El problema al escalar

  • ¿Cómo sabés qué nombre corresponde a qué saldo?
  • Si agregás una cuenta nueva, tenés que crear más variables sueltas
  • No hay forma de agrupar los datos con sus funciones relacionadas
  • Con 10 cuentas el código se vuelve difícil de mantener

La solución que propone POO

  • Agrupar datos y funciones en una sola unidad
  • Esa unidad se llama objeto
  • La descripción de cómo se construye esa unidad se llama clase
POO — Concepto fundamental

¿Qué es una Clase?

La clase es el molde, la plantilla, el plano

Una clase describe cómo van a ser los objetos que se creen a partir de ella. Define dos cosas:

  • Qué datos va a tener cada objeto (nombres de atributos)
  • Qué puede hacer cada objeto (métodos / comportamientos)

La clase en sí misma no representa ningún objeto concreto — es solo la descripción. No dice "el nombre es Ana", dice "habrá un atributo llamado nombre".

Analogías

Clase (el molde)Objeto (lo creado)
Plano de arquitecturaLa casa construida
Molde de galletitasCada galletita
Receta de cocinaEl plato preparado
Blueprint de un autoCada auto fabricado

Clave para entender

El plano no es la casa. El molde no es la galletita. La receta no es el plato. La clase no es el objeto — es lo que permite crear objetos.

Clase

¿Qué define una clase exactamente?

Atributos — los datos que va a tener

La clase dice "cada objeto de este tipo va a tener un nombre y una edad". No dice cuáles son los valores — eso lo decide cada objeto cuando se crea.

  • La clase define los nombres de los atributos
  • Cada objeto tendrá sus propios valores para esos atributos

Métodos — las acciones que puede realizar

La clase define qué acciones pueden realizar los objetos: saludar(), depositar(), calcular_promedio(). Todos los objetos de esa clase comparten esos métodos.

La clase Persona en pseudocódigo

CLASE Persona
  ATRIBUTOS:
    nombre  (texto)
    edad    (número)

  MÉTODOS:
    saludar()   → imprime un saludo
    cumpleaños() → incrementa la edad

  ← No hay valores aún.
     Solo la DESCRIPCIÓN de cómo
     serán los objetos Persona.

Importante

Cuando escribimos class Persona: en Python, no creamos ningún objeto todavía. Solo estamos definiendo el molde. Los objetos se crean después.

Objeto

¿Qué es un Objeto?

El objeto es la instancia concreta

Un objeto se crea a partir de una clase. En ese momento la clase toma vida: los atributos reciben valores reales, y el objeto ocupa espacio en memoria.

  • Estado — los valores reales de sus atributos
  • Comportamiento — puede ejecutar los métodos de su clase
  • Identidad — es único; dos objetos distintos aunque tengan los mismos valores

La relación clase → objeto

  • La clase dijo "habrá un atributo nombre"
  • El objeto dice "mi nombre es Ana"
  • La clase dijo "habrá un método saludar()"
  • El objeto lo ejecuta: "Hola, soy Ana"

Dos objetos de la misma clase

Atributoobjeto anaobjeto luis
nombre"Ana""Luis"
edad2025

Misma estructura (definida por la clase Persona), pero datos propios e independientes.

Terminología

  • Crear un objeto = instanciar una clase
  • Un objeto = una instancia de su clase
  • ana es una instancia de Persona
Clase vs Objeto

Clase vs Objeto — lado a lado

CLASE — el molde

  • Define los nombres de los atributos
  • Define los métodos disponibles
  • No tiene valores de datos propios
  • Se escribe una sola vez
  • No ocupa memoria de datos
  • Existe en el código fuente

OBJETO — la instancia

  • Tiene valores reales en sus atributos
  • Puede ejecutar los métodos de la clase
  • Tiene su propio estado independiente
  • Se pueden crear muchos de una misma clase
  • Ocupa memoria en tiempo de ejecución
  • Existe mientras el programa corre
Persona            ← CLASE: el molde (no tiene valores)
  atributos: nombre, edad
  métodos:   saludar()

ana  = Persona("Ana",  20)   ← OBJETO: instancia concreta con valores reales
luis = Persona("Luis", 25)   ← OBJETO: otro, independiente de ana
Clase vs Objeto

Una clase → muchos objetos

El poder del molde

De una sola clase se pueden crear todos los objetos que necesites, cada uno con sus propios datos. No tenés que redefinir la estructura cada vez — ya está en la clase.

  • Un molde de galletitas produce muchas galletitas
  • Un plano de arquitectura construye muchas casas
  • La clase Persona crea muchos objetos persona

Cada objeto es independiente

  • Modificar ana.edad no afecta a luis.edad
  • Cada objeto tiene su propio espacio en memoria
  • Comparten la estructura, no los datos
class Persona:
    nombre, edad, saludar()  ← un solo molde

# Creamos todos los objetos que queramos
ana    = Persona("Ana",    20)
luis   = Persona("Luis",   25)
carlos = Persona("Carlos", 30)
maria  = Persona("María",  22)

# Cada uno tiene sus propios datos
ana.saludar()    # "Hola, soy Ana"
carlos.saludar() # "Hola, soy Carlos"

# Modificar uno no afecta al otro
ana.edad = 21
print(luis.edad)  # 25 — no cambió
POO

¿Por qué usar este paradigma?

Lo que ganamos con clases y objetos

  • Organización — los datos y sus funciones viven juntos, no separados
  • Reutilización — una clase, infinitos objetos; herencia evita copiar
  • Mantenimiento — cambiar una clase no rompe el resto del sistema
  • Modelado natural — el código refleja cómo pensamos el problema
  • Escalabilidad — proyectos grandes con múltiples personas

¿Cuándo conviene usar POO?

  • Cuando tienes varias entidades con datos y comportamientos propios
  • Cuando necesitas crear muchas instancias similares (usuarios, productos...)
  • Cuando el proyecto va a crecer o trabajarán varias personas
  • Cuando querés modelar relaciones entre conceptos (herencia, composición)

Los 4 pilares que veremos

  • Encapsulamiento — ocultar el estado interno
  • Herencia — reutilizar código entre clases
  • Polimorfismo — mismo mensaje, distintas respuestas
  • Abstracción — mostrar solo lo necesario
Teoría

Los 4 pilares de POO

Primero el concepto, luego el código

Pilar 1 de 4

Encapsulamiento

¿Qué es?

Agrupar los datos y el comportamiento en una sola unidad, y proteger el estado interno del objeto del acceso directo desde afuera. La idea es separar qué hace el objeto de cómo lo hace.

¿Por qué importa?

  • Evita que el estado se corrompa accidentalmente
  • Fuerza que los cambios pasen por métodos controlados
  • Podés cambiar la implementación interna sin romper lo que usa el objeto

Analogía — el cajero automático

Presionás botones (interfaz pública) sin ver ni tocar los billetes internamente (estado protegido). No podés cambiar el saldo directamente — solo a través de las operaciones permitidas.

Sin — peligroso

cuenta.saldo = -999999

directo, sin ningún control

Con — seguro

cuenta.depositar(-100)

el método valida el monto

Lo veremos en código

Atributos _privados, __doble_guion, getters/setters, @property y diferencias con Java.

Pilar 2 de 4

Herencia

¿Qué es?

Una clase puede heredar atributos y métodos de otra clase. La clase hija recibe automáticamente todo lo de la clase padre, y puede agregar lo suyo o modificar lo que necesite.

¿Por qué importa?

  • Evita repetir código — principio DRY aplicado a clases
  • Expresa relaciones naturales: "un Perro ES un Animal"
  • Permite especializarse sin reescribir lo general

Analogía — biología

Un perro hereda de mamífero: respirar, sangre caliente, amamantar. No redefine todo eso — solo agrega lo suyo: ladrar, olfato agudo. La herencia en código funciona igual.

Jerarquías de ejemplo

Animal
 ├── Perro
 │     + ladrar()
 └── Gato
      + maullar()

Empleado
 ├── Gerente
 │     + departamento
 └── Analista
      + especialidad

Lo veremos en código

  • class Perro(Animal):
  • super().__init__() para reutilizar el constructor del padre
  • Sobreescribir (override) métodos
Pilar 3 de 4

Polimorfismo

¿Qué es?

Una clase hija puede reescribir (override) un método que heredó del padre. Cuando llamás ese método, Python ejecuta automáticamente la versión de la clase real del objeto — no la del padre.

El código que llama al método no necesita saber de qué tipo exacto es el objeto. Solo llama al método y Python hace lo correcto.

La clave — el override

  • El padre define el método: "todos los animales tienen sonido()"
  • Cada hijo lo reescribe con su propia implementación
  • Python decide en tiempo de ejecución cuál versión usar
  • Podés agregar nuevos tipos sin tocar el código existente

Analogía — "hablar"

Todos los seres vivos tienen el verbo "hablar", pero cada especie lo implementa diferente. El perro lo reescribe como "ladrar". El gato como "maullar". La persona como "conversar".

No hay un if es_perro → ladrar / elif es_gato → maullar. Cada uno tiene su propia versión del método.

Lo veremos en código

  • Clase padre define sonido() base
  • Cada clase hija reescribe sonido() con su comportamiento
  • Un for llama al mismo método en objetos distintos — cada uno responde diferente
Pilar 4 de 4

Abstracción

¿Qué es?

Mostrar al exterior solo lo necesario para usar el objeto, ocultando la complejidad interna. La interfaz pública es simple y clara; la implementación puede ser todo lo compleja que necesite.

¿Por qué importa?

  • El usuario del objeto no necesita entender los detalles internos
  • Podés cambiar la implementación sin afectar a quien usa la clase
  • Define qué puede hacer el objeto, no cómo lo hace

Abstracción vs Encapsulamiento

  • Abstracción → decidís qué mostrar al exterior
  • Encapsulamiento → el mecanismo para ocultar lo interno
  • Son complementarios, no son lo mismo

Analogía — manejar un auto

Usás el volante, el acelerador y el freno. No necesitás saber cómo funciona el motor, la transmisión ni el sistema de frenos. La interfaz es simple; el interior, complejo.

Ya usaste abstracción sin saberlo

Cuando llamás lista.append(x) no sabés cómo Python maneja la memoria internamente. Cuando usás print() no sabés cómo escribe en el terminal. Eso es abstracción.

Lo veremos en código

Al diseñar clases con interfaces claras: qué métodos exponemos, cuáles ocultamos, y por qué.

__init__, self, atributos y métodos

Clases

__init__ y self

__init__ — el constructor

  • Se ejecuta automáticamente al crear el objeto
  • Su trabajo: inicializar los atributos
  • Siempre recibe self como primer parámetro

self — el objeto mismo

  • Representa la instancia actual
  • Con self.atributo guardamos datos en el objeto
  • Sin self, la variable solo existe durante __init__
class Persona:
    def __init__(self, nombre: str, edad: int):
        self.nombre = nombre  # atributo de instancia
        self.edad   = edad

# Crear objetos
ana  = Persona("Ana", 20)
luis = Persona("Luis", 25)

# Acceder a atributos
print(ana.nombre)   # Ana
print(luis.edad)    # 25

# Cada objeto tiene sus propios valores
print(ana.nombre == luis.nombre)  # False
Clases

Métodos de instancia

¿Qué son?

  • Funciones dentro de una clase
  • Siempre reciben self como primer parámetro
  • Acceden a los atributos del objeto via self.x
  • Pueden retornar valores o solo hacer acciones

Llamar un método

objeto.metodo(args)

El self se pasa automáticamente — no lo escribes al llamar.

class Persona:
    def __init__(self, nombre: str, edad: int):
        self.nombre = nombre
        self.edad   = edad

    def saludar(self) -> None:
        print(f"Hola, soy {self.nombre}")

    def es_mayor_de_edad(self) -> bool:
        return self.edad >= 18

    def cumpleanos(self) -> None:
        self.edad += 1   # modifica el atributo

ana = Persona("Ana", 17)
ana.saludar()                     # Hola, soy Ana
print(ana.es_mayor_de_edad())    # False
ana.cumpleanos()
print(ana.es_mayor_de_edad())    # True
Clases

__str__ — representación legible

¿Qué es __str__?

  • Método especial que define cómo se muestra el objeto
  • Se llama automáticamente cuando usas print(objeto)
  • Debe retornar un string
  • Sin él, print muestra algo como <Persona object at 0x...>
class Persona:
    def __init__(self, nombre: str, edad: int):
        self.nombre = nombre
        self.edad   = edad

    def __str__(self) -> str:
        return f"Persona({self.nombre}, {self.edad})"

ana = Persona("Ana", 20)

# Sin __str__:  <__main__.Persona object at 0x10f...>
# Con __str__:
print(ana)              # Persona(Ana, 20)
print(str(ana))         # Persona(Ana, 20)
print(f"Alumna: {ana}") # Alumna: Persona(Ana, 20)

Módulo 03

Encapsulamiento

Proteger el estado interno de un objeto

Encapsulamiento

¿Qué significa que un atributo sea privado?

La idea detrás del encapsulamiento

Queremos que el estado interno de un objeto no sea modificable directamente desde afuera. Si el saldo de una cuenta bancaria se puede cambiar libremente, nada garantiza que sea válido.

  • El problema no es técnico — es de control
  • Queremos que los cambios pasen por métodos que validan
  • Si no hay control, cualquiera puede poner un saldo negativo directo

Lo que esperaríamos de "privado"

  • Que el lenguaje impida el acceso desde afuera de la clase
  • Que si alguien intenta acceder, reciba un error
  • Que la única forma de cambiar el valor sea a través de métodos controlados

El escenario sin encapsulamiento

class CuentaBancaria:
    def __init__(self, saldo):
        self.saldo = saldo  # público

c = CuentaBancaria(1000)

# Cualquiera puede hacer esto:
c.saldo = -99999    # sin error
c.saldo = "hola"    # sin error
c.saldo = None      # sin error

# El objeto quedó en estado inválido
# y Python no dijo nada

La pregunta clave

¿Cómo resuelven esto Java y Python? Spoiler: de formas muy distintas.

Encapsulamiento

Java — privado de verdad

Java

¿Cómo lo hace Java?

Java tiene modificadores de acceso que el compilador impone. No es una sugerencia, no es una convención — el lenguaje directamente no compila el código si violás la privacidad.

  • private → solo accesible dentro de la propia clase
  • protected → accesible desde la clase y sus subclases
  • public → accesible desde cualquier lado
  • Sin palabra clave → acceso a nivel de paquete
// Java
public class CuentaBancaria {
    private double saldo;  // REALMENTE privado

    public CuentaBancaria(double s) { saldo = s; }

    public double getSaldo() { return saldo; }

    public void depositar(double monto) {
        if (monto > 0) saldo += monto;
    }
}

// En otro archivo / clase:
CuentaBancaria c = new CuentaBancaria(1000);
c.getSaldo();       // ✓ OK — hay getter público
c.saldo = -100;    // ✗ ERROR DE COMPILACIÓN
// error: saldo has private access in CuentaBancaria

El compilador lo impide — no hay vuelta atrás

El código no compila. No se ejecuta. No hay forma de saltear esto en tiempo de ejecución sin usar Reflection (que es otra historia).

Encapsulamiento

Python — lo que realmente pasa

Python

El guion bajo _ es solo una convención

En Python, poner _saldo no hace nada técnico. Es equivalente a escribirle un cartel al programador que diga "por favor no toques esto". Pero si alguien decide ignorarlo, Python lo deja.

¿Qué significa en la práctica?

  • El IDE puede advertirte, pero el código corre igual
  • No hay error en tiempo de ejecución
  • No hay error de compilación (Python no compila)
  • El atributo está ahí, accesible, sin restricción real
class CuentaBancaria:
    def __init__(self, saldo: float):
        self._saldo = saldo  # _ = "no toques esto"

    def depositar(self, monto: float):
        if monto > 0:
            self._saldo += monto

c = CuentaBancaria(1000)
c.depositar(500)
print(c._saldo)      # 1500 — funciona sin error

# Violamos la "privacidad" — Python no dice nada
c._saldo = -99999   # funciona. Sin error. Sin advertencia.
print(c._saldo)      # -99999

# Python confía en que el programador sabe lo que hace

La diferencia es filosófica

Java no te deja. Python te deja pero te pide que no lo hagas. Son dos enfoques distintos sobre cuánto controlar al programador.

Encapsulamiento

Python — el doble guion __attr

¿Qué hace __saldo?

Python aplica name mangling: renombra internamente el atributo de __saldo a _CuentaBancaria__saldo. El objetivo es dificultar el acceso accidental desde fuera.

  • Acceder con c.__saldo da AttributeError
  • Pero podés acceder con c._CuentaBancaria__saldo
  • Sigue sin ser privado real — solo es más difícil de llegar

¿Cuándo usar cada uno?

  • _attr → la mayoría de los casos, atributo "interno"
  • __attr → cuando querés evitar colisiones en herencia
  • En Python real de producción: _attr es suficiente
class CuentaBancaria:
    def __init__(self, saldo: float):
        self.__saldo = saldo  # doble guion

c = CuentaBancaria(1000)

# Intento de acceso directo:
print(c.__saldo)
# AttributeError: 'CuentaBancaria' object has no attribute '__saldo'

# Pero Python solo lo renombró — sigue existiendo:
print(c._CuentaBancaria__saldo)   # 1000 — funciona igual

# Ver todos los atributos reales del objeto:
print(c.__dict__)
# {'_CuentaBancaria__saldo': 1000}

# Conclusión: no hay privado real en Python.
# El __ solo hace el acceso menos obvio.
Encapsulamiento

La filosofía de Python — "We're all adults here"

Python confía en el programador

Python no te impone restricciones de acceso porque asume que si estás usando la clase, sabés lo que hacés. La comunidad Python tiene un dicho:

"We're all consenting adults here"

Somos todos adultos que saben lo que hacen. No hace falta que el lenguaje nos proteja de nosotros mismos.

¿Entonces para qué sirve _ en Python?

Encapsulamiento

@property — getter elegante

¿Qué es @property?

  • Permite acceder a un atributo privado como si fuera público
  • El getter se llama con objeto.atributo (sin paréntesis)
  • El setter valida antes de asignar
  • Patrón más "Pythónico" que get_x()
class Temperatura:
    def __init__(self, grados: float):
        self._grados = grados

    @property
    def grados(self) -> float:
        return self._grados

    @grados.setter
    def grados(self, valor: float) -> None:
        if valor < -273.15:
            raise ValueError("Bajo cero absoluto")
        self._grados = valor

t = Temperatura(25.0)
print(t.grados)   # 25.0  ← sin paréntesis
t.grados = 30    # usa el setter
print(t.grados)   # 30
Módulo 04

Herencia

Una clase puede heredar atributos y métodos de otra

Herencia

Clase base y clase hija

¿Qué es herencia?

  • La clase hija recibe todos los atributos y métodos de la padre
  • Puede agregar nuevos atributos y métodos
  • Puede sobreescribir (override) métodos existentes
  • Evita repetir código — DRY aplicado a clases

Sintaxis

class Hija(Padre):

class Animal:
    def __init__(self, nombre: str):
        self.nombre = nombre

    def describir(self) -> None:
        print(f"Soy {self.nombre}")

    def sonido(self) -> str:
        return "..."

# Perro hereda de Animal
class Perro(Animal):
    def sonido(self) -> str:   # sobreescribe
        return "Guau"

p = Perro("Rex")
p.describir()           # Soy Rex  ← heredado
print(p.sonido())       # Guau     ← sobreescrito
Herencia

super() y extender el constructor

¿Para qué sirve super()?

  • Llama al método del padre desde la clase hija
  • En __init__: inicializa los atributos del padre sin repetir código
  • Luego la hija puede agregar sus propios atributos

Patrón típico

  1. Llama super().__init__(args)
  2. Agrega los atributos propios de la hija
class Empleado:
    def __init__(self, nombre: str, sueldo: float):
        self.nombre = nombre
        self.sueldo = sueldo

    def __str__(self) -> str:
        return f"{self.nombre} (${self.sueldo})"

class Gerente(Empleado):
    def __init__(self, nombre: str, sueldo: float, depto: str):
        super().__init__(nombre, sueldo)  # hereda init padre
        self.depto = depto   # atributo propio

    def __str__(self) -> str:
        return f"{self.nombre} — Gerente de {self.depto}"

g = Gerente("Luis", 5000, "Sistemas")
print(g)  # Luis — Gerente de Sistemas
Herencia

Polimorfismo — override en la práctica

¿Qué está pasando exactamente?

  • Animal define sonido() — es el método base
  • Perro lo reescribe con "Guau"
  • Gato lo reescribe con "Miau"
  • Cuando llamás a.sonido(), Python mira el tipo real de a y ejecuta la versión correcta

Por qué esto es poderoso

  • El for no sabe ni le importa si es Perro o Gato
  • Si agregás Pajaro mañana, el for no cambia
  • No hay if tipo == "perro" por ningún lado
class Animal:
    def __init__(self, nombre: str):
        self.nombre = nombre
    def sonido(self) -> str:
        return "..."   # versión base

class Perro(Animal):
    def sonido(self) -> str:  # reescribe
        return "Guau"

class Gato(Animal):
    def sonido(self) -> str:  # reescribe
        return "Miau"

animales = [Perro("Rex"), Gato("Luna"), Perro("Max")]

# El for no sabe qué tipo es cada uno
# Python sí — y ejecuta la versión correcta
for a in animales:
    print(f"{a.nombre}: {a.sonido()}")
# Rex: Guau
# Luna: Miau
# Max: Guau

isinstance() — verificar tipos

¿Qué es isinstance?

Es una función de Python que te dice si un objeto es instancia de una clase determinada. No tiene nada que ver con el polimorfismo — es simplemente una herramienta para verificar tipos en tiempo de ejecución.

  • Retorna True si el objeto es de esa clase o de una clase hija
  • Útil cuando necesitás tomar decisiones basadas en el tipo real del objeto
  • Si lo usás mucho, puede ser señal de que el diseño necesita más polimorfismo
# isinstance verifica el tipo del objeto
rex = Perro("Rex")

isinstance(rex, Perro)   # True  — es un Perro
isinstance(rex, Animal)  # True  — Perro hereda de Animal
isinstance(rex, Gato)    # False — no es un Gato

# Ejemplo de uso válido:
def procesar(animal):
    if isinstance(animal, Perro):
        animal.buscar_pelota()  # método solo de Perro
    animal.sonido()             # esto ya es polimorfismo

# El polimorfismo va en .sonido() — no en el isinstance
Composición

Concatenar métodos entre objetos

¿Qué es esto?

Un objeto puede usar el resultado de otro objeto como argumento. No es magia — es simplemente que un método retorna un valor y ese valor se pasa a otro método.

¿Cómo leerlo?

obj1.metodo( obj2.metodo(x, y) )

  1. Primero se ejecuta lo de adentro: obj2.metodo(x, y)
  2. Ese resultado se usa como argumento de obj1.metodo(...)

Es lo mismo que hacer

resultado = obj2.metodo(x, y)
obj1.metodo(resultado)

La versión concatenada es un atajo — pero hacen exactamente lo mismo.

class Calculadora:
    def sumar(self, a: float, b: float) -> float:
        return a + b

class Formateador:
    def redondear(self, valor: float) -> str:
        return f"Resultado: {round(valor, 2)}"

calc = Calculadora()
fmt  = Formateador()

# Concatenado — un objeto usa el resultado del otro
print(fmt.redondear( calc.sumar(3.141, 2.718) ))
# Resultado: 5.86

# Es equivalente a hacerlo en dos pasos:
total = calc.sumar(3.141, 2.718)   # 5.859
texto = fmt.redondear(total)        # "Resultado: 5.86"
print(texto)

Ejemplo práctico — Producto + Carrito

El patrón

Cada clase tiene su responsabilidad. Producto sabe calcular su precio con IVA. Carrito sabe aplicar descuentos. Ninguna necesita saber cómo funciona la otra — solo usa su resultado.

Paso a paso

  1. p.con_iva() retorna 115.0
  2. Ese 115.0 entra como argumento de c.aplicar_descuento(115.0, 0.10)
  3. Resultado final: 103.5

Clave

Lo importante es que el método interno retorne un valor. Si no hay return, no hay nada que pasar al método externo.

class Producto:
    def __init__(self, nombre: str, precio: float):
        self.nombre = nombre
        self.precio = precio

    def con_iva(self) -> float:
        return self.precio * 1.15

class Carrito:
    def aplicar_descuento(self, total: float,
                         pct: float) -> float:
        return total * (1 - pct)

p = Producto("Laptop", 100)
c = Carrito()

# Concatenado:
final = c.aplicar_descuento( p.con_iva(), 0.10 )
print(final)  # 103.5

# Equivalente en dos pasos:
precio_iva = p.con_iva()              # 115.0
final = c.aplicar_descuento(precio_iva, 0.10)
print(final)  # 103.5

class CuentaBancaria:
    def __init__(self, titular: str, saldo: float):
        self.titular = titular
        self._saldo  = saldo

    def depositar(self, monto: float) -> None:
        if monto > 0:
            self._saldo += monto

    def retirar(self, monto: float) -> bool:
        if monto > self._saldo:
            return False
        self._saldo -= monto
        return True

    def get_saldo(self) -> float:
        return self._saldo

    def __str__(self) -> str:
        return f"{self.titular} — Saldo: ${self._saldo}"
# Prueba
c = CuentaBancaria("Ana", 1000)
c.depositar(500)
print(c.get_saldo())      # 1500

ok = c.retirar(200)
print(ok)                  # True
print(c.get_saldo())      # 1300

fallido = c.retirar(9999)
print(fallido)             # False
print(c)                   # Ana — Saldo: $1300

Puntos clave

  • retirar retorna bool — el llamador sabe si funcionó
  • _saldo solo se modifica dentro de la clase

Estudiante con notas

class Estudiante:
    def __init__(self, nombre: str):
        self.nombre  = nombre
        self._notas  = []

    def agregar_nota(self, nota: float) -> None:
        self._notas.append(nota)

    def promedio(self) -> float:
        if not self._notas:
            return 0
        return sum(self._notas) / len(self._notas)

    def aprobo(self) -> bool:
        return self.promedio() >= 6

    def __str__(self) -> str:
        return f"{self.nombre} — Promedio: {self.promedio():.1f}"
# Prueba
e = Estudiante("Ana")
for nota in [8, 9, 7, 10]:
    e.agregar_nota(nota)

print(e.promedio())  # 8.5
print(e.aprobo())    # True
print(e)             # Ana — Promedio: 8.5

e2 = Estudiante("Luis")
e2.agregar_nota(3)
print(e2.aprobo())   # False

Puntos clave

  • aprobo() reutiliza promedio() — DRY
  • _notas = [] en __init__ evita el bug de lista compartida

Jerarquía de vehículos

class Vehiculo:
    def __init__(self, marca: str, vel_max: int):
        self.marca    = marca
        self.vel_max  = vel_max

    def describir(self) -> str:
        return f"{self.marca} — {self.vel_max} km/h"

class Auto(Vehiculo):
    def __init__(self, marca, vel_max, puertas: int):
        super().__init__(marca, vel_max)
        self.puertas = puertas

    def describir(self) -> str:
        return f"{self.marca} — {self.puertas} puertas"

class Moto(Vehiculo):
    def __init__(self, marca, vel_max, tipo: str):
        super().__init__(marca, vel_max)
        self.tipo = tipo

    def describir(self) -> str:
        return f"{self.marca} ({self.tipo})"
vehiculos = [
    Auto("Toyota", 180, 4),
    Moto("Honda",  220, "deportiva"),
    Auto("Ford",   160, 2),
]

for v in vehiculos:
    print(v.describir())
# Toyota — 4 puertas
# Honda (deportiva)
# Ford — 2 puertas

print(isinstance(vehiculos[0], Vehiculo))  # True

Puntos clave

  • super().__init__ evita repetir self.marca = marca
  • Polimorfismo: mismo describir(), resultados distintos
