La Cocina — el Servidor

API = Application Programming Interface — es el contrato entre dos programas. Define qué podés pedir, cómo pedirlo y qué formato te responde. Hoy vemos el tipo más común: las APIs que usan HTTP y responden JSON.

La base de internet

Cómo se hablan cliente y servidor

El cliente

  • Siempre inicia la comunicación
  • Manda un Request (pedido)
  • Puede ser tu navegador, curl, Postman, o Python

El servidor

  • Escucha y responde cuando le piden
  • Manda una Response (respuesta)
  • Un archivo, datos JSON, un error — lo que corresponda

🌐 restcountries.com

Cada vez que usás una API esto es lo que pasa. Vos mandás un mensaje (Request) por HTTP. El servidor te responde (Response). Vamos a diseccionar ambos.

El protocolo

HTTP — el idioma de la web

¿Qué es HTTP?

HyperText Transfer Protocol. Es el idioma que usan cliente y servidor para hablar. Define exactamente cómo es el formato de un pedido y de una respuesta.

Cuando escribís una URL en el navegador, el navegador arma un mensaje HTTP y lo manda. Cuando tu Python usa requests.get(), hace exactamente lo mismo.

¿Qué es HTTPS?

HTTP con el canal encriptado (TLS). Mismo protocolo, diferente que nadie intercepte la conversación. Todas las APIs serias usan HTTPS.

Un mensaje HTTP tiene dos partes

REQUEST (lo que vos mandás)
GET /v3.1/name/argentina HTTP/1.1
Host: restcountries.com
Accept: application/json
RESPONSE (lo que el servidor te manda)
HTTP/1.1 200 OK
Content-Type: application/json

[{ "name": { "common": "Argentina" }, ... }]
El verbo de la request

Los métodos HTTP — qué le estás pidiendo al servidor

Los 4 más usados

Método ¿Qué le decís al servidor? Acción
GET Dame este recurso Leer
POST Creá esto nuevo Crear
PUT Reemplazá esto Actualizar
DELETE Borrá esto Borrar

Hoy usamos solo GET. Las APIs de lectura de datos casi siempre son todo GET.

GET — solo lee, nunca modifica

  • No manda cuerpo (body) — el pedido va en la URL
  • Podés repetirlo mil veces, el servidor no cambia
  • Es lo que hace el navegador cuando escribís una URL
  • Tiene query params: ?lang=es&page=2

¿Cuándo necesitás POST?

Cuando mandás datos al servidor: crear un usuario, postear un tweet, subir un archivo. Ahí el cuerpo de la request tiene el contenido que mandás.

Hoy no llegamos a esto — queda para cuando trabajemos con APIs que requieren autenticación.

Lo que te devuelve el servidor

Anatomía de una respuesta HTTP

Tres partes siempre

① Status line:
HTTP/1.1 200 OK
② Headers:
Content-Type: application/json
Content-Length: 3842
③ Body (el contenido real):
[{ "name": { ... }, ... }]

① El status code

Un número de 3 dígitos que dice si salió bien o mal. El primer dígito es la categoría:

2xxÉxito — todo bien
3xxRedirección — fijate allá
4xxError tuyo — pediste mal
5xxError del servidor — falló él

② Headers y ③ Body

  • Headers: metadatos — qué tipo de contenido viene, cuánto pesa, cookies, etc.
  • Body: el contenido real — el JSON que tu Python va a leer

El estilo de diseño

¿Qué hace que una API sea "REST"?

REST no es un protocolo — es una convención

REST = Representational State Transfer. Es un conjunto de reglas de diseño para que las APIs sean predecibles.

  • Los recursos son sustantivos en la URL: /countries, /users
  • Las acciones son los métodos HTTP: GET, POST, PUT, DELETE
  • Cada request es independiente — el servidor no recuerda la anterior
  • Responde con JSON (el estándar moderno)

URLs REST vs URLs no-REST

/getCountry?name=arg ✗ Acción en la URL — no REST
/name/argentina ✓ Recurso en la URL, método GET implícito
/deleteUser?id=5 ✗ Acción en la URL — no REST
DELETE /users/5 ✓ Recurso en URL, DELETE como método

Para consumidores de API (que es lo que hacemos hoy) lo que importa es esto: cada endpoint es una URL a la que le hacés un GET y te devuelve JSON.

Mismo pedido, tres herramientas

La misma request desde el navegador, curl y Python

Navegador

Escribís la URL y Enter. El navegador arma el GET, te muestra la respuesta.

restcountries.com/v3.1/name/argentina

Ves el JSON crudo. Probalo ahora.

curl — terminal

Herramienta de línea de comando. Más control, misma idea.

curl -s \
 restcountries.com/v3.1/\
 name/argentina

El -s es "silent" — sin barra de progreso.

Python + requests

Desde código. El resultado ya es un dict.

import requests

r = requests.get(
  "...name/argentina"
)
data = r.json()

Las tres hacen exactamente lo mismo: arman un HTTP GET con esa URL y procesan la respuesta. La diferencia es solo qué tan fácil es trabajar con el resultado en código.

La librería

requests — HTTP para humanos

Instalación (una sola vez)

Los métodos principales

Función¿Para qué?
requests.get(url)HTTP GET
requests.post(url, json=data)HTTP POST con cuerpo JSON
requests.put(url, json=data)HTTP PUT
requests.delete(url)HTTP DELETE

Todos devuelven un objeto Response.

El objeto Response — qué tiene

Atributo / Método¿Qué contiene?
r.status_codeNúmero: 200, 404, 500…
r.okTrue si status < 400
r.textCuerpo como string crudo
r.json()Cuerpo parseado a dict/list
r.headersHeaders de la respuesta
r.raise_for_status()Lanza excepción si 4xx/5xx
El patrón que vas a usar siempre

get → raise_for_status → json

el patrón base
import requests

url = "https://restcountries.com/v3.1/name/argentina"

r = requests.get(url, timeout=5)   # 1
r.raise_for_status()                 # 2
data = r.json()                       # 3

print(type(data))  # <class 'list'>
print(data[0].keys())

Por qué esas 3 líneas en ese orden

1 · .get(timeout=5) ← Manda el pedido. timeout obligatorio: sin él, si el servidor no responde tu programa se congela para siempre
2 · raise_for_status() ← Si el status es 4xx o 5xx, lanza una excepción. Sin esto tu código sigue aunque el servidor haya fallado
3 · .json() ← Convierte el texto JSON del cuerpo a un dict o list de Python

Si no ponés raise_for_status() y el server responde 404, r.json() va a intentar parsear el cuerpo del error. Podés llevarte sorpresas.

Live coding — paso 1

Función get_country(name)

paises.py
import requests

BASE = "https://restcountries.com/v3.1"

def get_country(name: str) -> dict:
    url = f"{BASE}/name/{name}"
    r = requests.get(url, timeout=5)
    r.raise_for_status()
    return r.json()[0]     # primer resultado


# probamos
arg = get_country("argentina")
print(arg["name"]["common"])
print(arg["capital"][0])
print(arg["population"])
salida
Argentina
Buenos Aires
45376763

Por qué [0] al final

El endpoint /name/ devuelve una lista, aunque busques un solo país. Puede haber varios resultados si el nombre coincide parcialmente. Con [0] agarramos el primero.

Ahora agregamos una función para mostrarlo

def mostrar(p: dict):
    nombre    = p["name"]["common"]
    capital   = p.get("capital", ["—"])[0]
    region    = p.get("region", "—")
    poblacion = p.get("population", 0)
    area      = p.get("area", 0)
    densidad  = poblacion / area if area else 0

    print(f"{nombre} ({region})")
    print(f"  Capital:  {capital}")
    print(f"  Población: {poblacion:,}")
    print(f"  Densidad:  {densidad:.2f} hab/km²")

Ordenar países por población

paises.py — ordenar
paises = get_region("south america")

def get_poblacion(p):
    return p.get("population", 0)

ordenados = sorted(
    paises,
    key=get_poblacion,
    reverse=True,
)

print("Ranking por población:\n")
for i, p in enumerate(ordenados, 1):

¿Cómo le decís a sorted por qué campo ordenar?

sorted acepta un parámetro key que es una función. Esa función recibe un elemento de la lista y devuelve el valor a comparar.

Definís la función con def y la pasás por nombre — sin paréntesis, no la estás llamando, se la estás entregando a sorted.

# 1) definís la función
def get_poblacion(p):
    return p.get("population", 0)

# 2) sorted la llama por cada elemento
#    internamente hace: get_poblacion(p)
#    para cada p de la lista
sorted(paises, key=get_poblacion, reverse=True)

Dato importante: .get("campo", 0)

Algunos países no tienen todos los campos. .get() con un default evita el KeyError sin que rompas nada.

Qué puede salir mal — y cómo manejarlo

try / except con requests

Tres casos reales

SituaciónExcepción
Sin internet / DNS fallaConnectionError
País no encontrado (404)HTTPError
Servidor no respondeTimeout

Las tres viven en requests.exceptions.

versión con manejo de errores
from requests.exceptions import (
    HTTPError, ConnectionError, Timeout
)

def get_country(name):
    url = f"{BASE}/name/{name}"
    try:
        r = requests.get(url, timeout=5)
        r.raise_for_status()
        return r.json()[0]
    except Timeout:
        print("⏱ La API tardó demasiado")
    except ConnectionError:
        print("📡 Sin conexión a internet")
    except HTTPError as e:
        print(f"❌ Error {e.response.status_code}: no encontrado")
    return None

El orden importa: poné primero las excepciones más específicas (Timeout, HTTPError) antes de las más generales. Si atrapás Exception primero, las demás nunca se alcanzan.

Tarea: subir
la dificultad con OOP

El mismo código — pero encapsulado en clases

Consigna

Reescribe el ejercicio usando clases

Clase Country

En vez de pasar dicts por todos lados, modela un país como objeto:

  • __init__(self, data: dict) — recibe el dict crudo de la API y extrae los atributos (nombre, capital, poblacion, area, region)
  • __str__(self) — devuelve una cadena legible con todos los datos del país
  • density(self) — calcula y devuelve habitantes por km²
  • comparar(self, otros: list) — imprime una tabla comparativa e indica el ganador en cada categoría

Clase CountryAPI

Encapsula todas las requests a la API:

  • by_name(name) → devuelve un objeto Country
  • by_region(region) → devuelve list[Country]
  • El manejo de errores va dentro de la clase — el main no sabe nada de requests

Cómo debe quedar el main

# Sin clases (hoy):
paises = get_region("south america")
for p in paises:
    mostrar(p)

# Con clases (parcial):
api = CountryAPI()
paises = api.by_region("south america")
for p in paises:
    print(p)

El main queda limpio. La lógica de cómo se pide y cómo se muestra vive en las clases.
