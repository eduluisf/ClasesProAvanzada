Aislamiento de proyectos

Modulo 01

El problema

¿Por qué no podemos simplemente hacer pip install y ya?

El problema

¿Qué pasa sin entornos virtuales?

Escenario real

Imagina que tienes dos proyectos en tu computadora:

  • Proyecto A usa requests==2.25.0
  • Proyecto B usa requests==2.31.0
  • Ambos usan la misma libreria pero versiones diferentes

Sin venv: conflicto garantizado

  • Python tiene una sola carpeta global para paquetes
  • Si instalas requests 2.31, sobreescribes la 2.25
  • El Proyecto A se rompe silenciosamente
# Sin venv — todo se instala globalmente

$ pip install requests==2.25.0
# Proyecto A funciona bien

$ pip install requests==2.31.0
# Proyecto B funciona bien
# Proyecto A ahora esta ROTO
# porque requests 2.25 ya no existe

$ pip list
requests   2.31.0  # solo una version

El problema de fondo

Sin aislamiento, todos tus proyectos comparten las mismas dependencias. Cambiar una libreria en un proyecto puede romper otro.

Analogia

Entorno virtual = caja independiente

Sin venv: un solo escritorio

Todos tus proyectos comparten las mismas herramientas. Si cambias una, afectas a todos.

Es como tener una sola caja de herramientas para toda la casa: si prestas el martillo, nadie mas lo puede usar.

Con venv: cada proyecto tiene su caja

Cada proyecto tiene sus propias dependencias, su propia version de Python, y sus propias librerias. Nada se mezcla.

Comparacion visual

Sin venvCon venv
1 carpeta global de paquetes1 carpeta por proyecto
Conflictos entre versionesVersiones independientes
Instalas algo y afectas todoSolo afecta ese proyecto
Dificil de reproducirFacil de compartir y replicar

Regla de oro

Nunca instales paquetes en el Python global. Siempre usa un entorno virtual. Es la primera cosa que haces al empezar un proyecto.

Modulo 02

Crear y usar un venv

Paso a paso desde la terminal

Paso 1

Crear el entorno virtual

El comando

python -m venv crea una carpeta con una copia aislada de Python y pip. El nombre mas comun es venv o .venv.

# Crear el entorno virtual
$ python -m venv venv

# Esto crea una carpeta llamada "venv" con:
#   venv/
#   ├── bin/          (Scripts: python, pip)
#   ├── include/      (Headers de C)
#   ├── lib/          (Paquetes instalados)
#   └── pyvenv.cfg   (Configuracion)

¿Que es -m venv?

  • -m significa "ejecutar este modulo"
  • venv es un modulo que viene con Python 3
  • No necesitas instalar nada extra

Nombres comunes

  • venv — el mas clasico y simple
  • .venv — con punto, queda oculto en Linux/Mac
  • Ambos funcionan exactamente igual

Importante

La carpeta venv/ no se sube al repositorio. Se agrega al .gitignore. Solo contiene binarios — no es codigo fuente.

Paso 2

Activar el entorno virtual

¿Por que hay que activarlo?

Crear el venv no lo activa automaticamente. Activarlo le dice a tu terminal: "usa el Python y pip de esta carpeta, no el global".

# macOS / Linux
$ source venv/bin/activate

# Windows (CMD)
$ venv\Scripts\activate

# Windows (PowerShell)
$ venv\Scripts\Activate.ps1

¿Como se que esta activo?

Aparece el nombre del venv en la terminal:

(venv) $ _
# Este prefijo indica que estas
# trabajando dentro del entorno virtual

¿Que cambia al activar?

  • python apunta al Python del venv, no al global
  • pip instala paquetes dentro del venv
  • Los paquetes globales no se ven desde el venv
# Verificar que estas usando el correcto
$ which python
/mi-proyecto/venv/bin/python

$ which pip
/mi-proyecto/venv/bin/pip

# Si NO tienes venv activo:
$ which python
/usr/bin/python3  # global

Desactivar

Simplemente escribe deactivate en la terminal. Vuelves al Python global.

(venv) $ deactivate
$ _  # ya no hay prefijo
Paso 3

Instalar paquetes con pip

pip — el gestor de paquetes de Python

pip install descarga e instala librerias desde PyPI (Python Package Index). Con el venv activo, todo se instala dentro de la carpeta del entorno.

# Asegurate de tener el venv activo
(venv) $ pip install requests
(venv) $ pip install flask
(venv) $ pip install pandas==2.1.0

# Ver que hay instalado
(venv) $ pip list
Package    Version
---------- -------
flask      3.0.0
pandas     2.1.0
requests   2.31.0
pip        23.3.1

Comandos utiles de pip

Comando¿Que hace?
pip install paqueteInstala la ultima version
pip install paquete==1.2Instala version especifica
pip uninstall paqueteDesinstala un paquete
pip listLista paquetes instalados
pip show paqueteDetalles de un paquete

Clave

Si haces pip install sin el venv activo, el paquete se instala globalmente y afecta a todos tus proyectos. Siempre verifica el prefijo (venv).

Modulo 03

requirements.txt

Compartir dependencias con tu equipo

requirements.txt

Guardar y restaurar dependencias

¿Que es requirements.txt?

Es un archivo de texto que lista todos los paquetes y sus versiones que tu proyecto necesita. Es la forma estandar de compartir dependencias.

# Generar el archivo con las dependencias actuales
(venv) $ pip freeze > requirements.txt

# Contenido de requirements.txt:
flask==3.0.0
pandas==2.1.0
requests==2.31.0
numpy==1.26.2
Jinja2==3.1.2

Este archivo SI se sube al repositorio

A diferencia de la carpeta venv/, el requirements.txt es parte del codigo fuente. Permite que cualquier persona recree el entorno.

Restaurar dependencias (otro dev o servidor)

Cuando alguien clona tu proyecto, recrea el entorno asi:

# 1. Clonar el proyecto
$ git clone mi-proyecto
$ cd mi-proyecto

# 2. Crear un venv nuevo
$ python -m venv venv

# 3. Activarlo
$ source venv/bin/activate

# 4. Instalar TODAS las dependencias
(venv) $ pip install -r requirements.txt

# Listo — tiene exactamente las mismas
# versiones que el proyecto original

Flujo completo

  1. Crear venv
  2. Activar venv
  3. Instalar paquetes con pip
  4. pip freeze > requirements.txt
  5. Subir requirements.txt al repo
Modulo 04

¿Cuando usar venv?

Casos donde SI conviene y donde NO es necesario

Cuando SI usar venv

Escenarios donde es obligatorio

Proyectos con dependencias externas

  • Usas pip install para instalar algo
  • Tu proyecto depende de flask, django, pandas, etc.
  • Necesitas una version especifica de una libreria

Trabajo en equipo

  • Otro desarrollador va a clonar tu proyecto
  • Necesitan las mismas versiones exactas
  • requirements.txt + venv lo resuelve

Multiples proyectos en la misma PC

  • Proyecto A usa Django 4.2
  • Proyecto B usa Django 5.0
  • Sin venv, uno de los dos se rompe

Deploy a produccion

  • El servidor necesita saber que instalar
  • requirements.txt lo documenta
  • Garantiza que produccion tiene las mismas versiones que desarrollo

En resumen

Si tu proyecto usa cualquier paquete externo, usa venv. Punto.

Cuando NO se necesita

Escenarios donde no es necesario

Scripts simples sin dependencias

  • Tu script solo usa la libreria estandar de Python
  • Modulos como os, sys, math, json, datetime
  • No necesitas pip install para nada
# Este script NO necesita venv
import os
import json
import math

# Todo viene con Python, no hay
# dependencias externas
radio = 5
area = math.pi * radio ** 2
print(f"Area: {area}")

Experimentacion rapida en REPL

  • Estas probando algo rapido en el interprete
  • No es un proyecto formal, solo explorar
  • Aun asi, no instales paquetes globalmente

Ojo: no es excusa

Que un script sea "pequeno" no significa que no necesite venv. Si usas pip install, usa venv. El tamano del proyecto no importa — importa si tienes dependencias externas.

Modulo 05

.gitignore

Que subir y que NO subir al repositorio

.gitignore

Configurar el .gitignore

¿Que es .gitignore?

Un archivo que le dice a Git que archivos o carpetas ignorar. Todo lo que este ahi no se sube al repositorio.

# .gitignore para proyectos Python

# Entorno virtual
venv/
.venv/

# Cache de Python
__pycache__/
*.pyc

# Variables de entorno
.env

# IDE
.vscode/
.idea/

¿Por que NO subir venv/?

  • Pesa mucho (cientos de MB con paquetes)
  • Contiene binarios compilados para TU sistema
  • Otro dev en Windows no puede usar tu venv de Mac
  • requirements.txt es suficiente para recrearlo

¿Que SI subir?

  • requirements.txt — lista de dependencias
  • .gitignore — para que todos lo respeten
  • Tu codigo fuente (.py)
  • Archivos de configuracion del proyecto

Estructura tipica de un proyecto

mi-proyecto/
├── venv/              ← NO se sube
├── __pycache__/       ← NO se sube
├── .gitignore         ← SI se sube
├── requirements.txt   ← SI se sube
├── main.py            ← SI se sube
└── app/               ← SI se sube

Ejemplo completo

Crear un proyecto desde cero con venv

Ejemplo completo

Proyecto desde cero

Paso 1: Crear la estructura

# Crear carpeta del proyecto
$ mkdir mi-api
$ cd mi-api

# Crear el entorno virtual
$ python -m venv venv

# Activarlo
$ source venv/bin/activate
(venv) $

Paso 2: Instalar dependencias

(venv) $ pip install flask requests

# Guardar dependencias
(venv) $ pip freeze > requirements.txt

Paso 3: Crear el .gitignore

# Crear .gitignore
$ echo "venv/" > .gitignore
$ echo "__pycache__/" >> .gitignore

Paso 4: Escribir codigo

# main.py
from flask import Flask

app = Flask(__name__)

@app.route("/")
def hola():
    return "Hola desde mi API!"

if __name__ == "__main__":
    app.run(debug=True)

Paso 5: Ejecutar

(venv) $ python main.py
# Running on http://127.0.0.1:5000
Errores comunes

Errores que todos cometen

1. Olvidar activar el venv

Instalas paquetes pensando que estas en el venv, pero en realidad van al Python global.

# MAL — sin (venv) al inicio
$ pip install flask
# Se instala GLOBALMENTE

# BIEN — con el venv activo
(venv) $ pip install flask
# Se instala en el venv

2. Subir venv/ al repositorio

Hace que el repo pese cientos de MB y no sirve en otra computadora. Siempre usa .gitignore.

3. No generar requirements.txt

Tu proyecto funciona en tu PC pero nadie mas puede instalarlo porque no saben que paquetes necesita.

4. Editar archivos dentro de venv/

La carpeta venv es generada automaticamente. Nunca edites nada ahi. Si algo se rompe, borrala y creala de nuevo.

# Si el venv se corrompe:
$ rm -rf venv
$ python -m venv venv
$ source venv/bin/activate
(venv) $ pip install -r requirements.txt
# Como nuevo
