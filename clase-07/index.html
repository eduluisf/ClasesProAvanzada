La pesadilla del programador sin Git

  • proyecto_final_v2_bueno.py

¿Cual era el bueno? ¿Que cambio entre versiones? ¿Como volviste atras cuando algo se rompio?

Lo que resuelve Git

  • Historial completo — cada cambio guardado con fecha y autor
  • Volver atras — deshacer cualquier cambio sin perder nada
  • Trabajar en equipo — varios en el mismo proyecto sin pisarse
  • Experimentar sin miedo — ramas para probar cosas nuevas

Git ≠ GitHub

Git es la herramienta que corre en tu computadora. GitHub es un sitio web para guardar repositorios Git en la nube y colaborar. Puedes usar Git sin GitHub.

Como funciona

Git guarda fotos del proyecto, no diferencias

Cada commit = una foto completa

Cada vez que hacés git commit, Git guarda una instantanea (snapshot) de todos los archivos en ese momento. Si un archivo no cambio, Git no lo duplica — apunta a la version anterior.

Analogia: el videojuego

Un commit es como guardar la partida. Podes seguir jugando, probar cosas, morir, y siempre volver al ultimo punto guardado. El historial de commits es tu lista de saves.

Las tres areas

Working Directory · Staging · Repositorio

1. Working Directory

Tu carpeta de trabajo normal. Aqui editas archivos. Git ve los cambios pero aun no los guarda.

Archivos: modificados o sin trackear

2. Staging Area

La "sala de espera" antes del commit. Con git add preparas exactamente que cambios van a entrar al proximo commit.

Archivos: listos para commitear

3. Repositorio

La base de datos local de Git (carpeta .git/). Aqui viven todos los commits y el historial completo.

Archivos: guardados permanentemente

El flujo

Editas archivo  →  git add archivo.py  →  git commit -m "mensaje"

Modulo 02

Comandos esenciales

Los 10 comandos que usas todos los dias

Primeros pasos

Configurar e iniciar un repositorio

Configurar tu identidad — una sola vez

# Quien sos (aparece en cada commit)
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

Iniciar un repo nuevo

# En la carpeta de tu proyecto:
git init

# Crea la carpeta .git/ oculta
# donde Git guarda todo

Clonar un repo existente

# Bajar un repo de GitHub
git clone https://github.com/usuario/repo.git

# Crea una carpeta con el proyecto
# y toda la historia de commits

¿init o clone?

init → proyecto nuevo en tu maquina que vas a subir a GitHub despues.
clone → proyecto que ya existe en GitHub (tuyo o de otro).

Ver el estado

git status y git diff

git status — ¿que esta pasando?

$ git status

On branch main
Changes not staged for commit:
  modified:   app.py

Untracked files:
  nuevo.py

nothing added to commit but untracked files present

¿Que me dice?

  • modified — archivo existente con cambios sin agregar
  • untracked — archivo nuevo que Git no conoce todavia
  • staged — listo para el proximo commit

git diff — ¿que cambio exactamente?

# Ver cambios sin agregar al staging
git diff

# Ver cambios ya en staging
git diff --staged

# Resultado: lineas quitadas/agregadas
- print("hola")
+ print("hola mundo")

Regla practica

Antes de cada commit: git status para ver el panorama, git diff para revisar los cambios. Commitear sin revisar es como enviar un email sin leerlo.

Guardar cambios

git add y git commit

git add — preparar cambios

# Agregar un archivo especifico
git add app.py

# Agregar varios archivos
git add app.py utils.py

# Agregar TODOS los cambios
git add .

# Agregar partes de un archivo
git add -p app.py

Cuidado con git add .

Agrega todo incluyendo archivos que no queres versionar (claves, .env, __pycache__). Usa un archivo .gitignore para decirle a Git que ignorar.

git commit — guardar la foto

# Commit con mensaje
git commit -m "Agrega funcion de login"

# Add + commit en un paso
# (solo para archivos ya trackeados)
git commit -am "Corrige bug en validacion"

Buenos mensajes de commit

El mensaje debe explicar que cambio y (si no es obvio) por que. Usa el imperativo: "Agrega login", "Corrige bug en X", "Refactoriza funcion Y". No: "cambios", "aaa", "fix".

Historial y exclusiones

git log y .gitignore

git log — ver la historia

# Historia completa
git log

# Una linea por commit
git log --oneline

# Con grafico de ramas
git log --oneline --graph --all

# Buscar commits de un autor
git log --author="Eduardo"

# Ver cambios de cada commit
git log -p

.gitignore — que NO versionar

# archivo: .gitignore

# Entornos virtuales
venv/
.env

# Python compilado
__pycache__/
*.pyc

# IDEs
.vscode/
.idea/

# Archivos de sistema
.DS_Store
Thumbs.db

Regla de oro

Nunca commitees claves, tokens, passwords o .env. Una vez en el historial, son publicos para siempre aunque los borres despues.

Deshacer cambios

Como volver atras sin entrar en panico

Descartar cambios en un archivo

# El archivo tenia cambios,
# quiero volver a como estaba
git restore app.py

# Quitar del staging (pero mantener cambios)
git restore --staged app.py

Ver y volver a un commit anterior

# Ver el codigo en un commit pasado
git checkout a3f9c12

# Volver al presente
git checkout main

Deshacer el ultimo commit

# Deshace el commit pero mantiene los cambios
# como si nunca lo hubieras commitado
git reset --soft HEAD~1

# Deshace el commit Y quita los cambios
# del staging (archivos quedan modificados)
git reset HEAD~1

# Deshace el commit Y borra los cambios
# ⚠ NO reversible
git reset --hard HEAD~1

Tip clave

Si ya hiciste push, no uses reset. Usa git revert que crea un nuevo commit que deshace el anterior sin reescribir la historia.

Modulo 03

Ramas (Branches)

Trabajar en paralelo sin romper lo que funciona

Branches

¿Que es una rama?

Una rama = una linea de desarrollo

Una rama es como un universo paralelo de tu proyecto. Podes hacer cambios en la rama sin afectar a las demas. Cuando estes listo, las unís (merge).

  • main — la rama principal, codigo estable
  • Ramas nuevas para cada feature o bug
  • Multiples personas en ramas distintas = sin conflictos

Analogia: cuadernos de borrador

Tu cuaderno principal (main) tiene el trabajo final. Cuando queres probar algo, sacas un cuaderno de borrador (branch). Si queda bien, pásalo al principal. Si no, tiralo sin afectar el original.

Ramas en el grafico

●
c91d — Primer commit
●
7b2e — Agrega estilos
●
a3f9 — Loginmain
●
f10c — Nueva UIfeature/ui

Ambas ramas parten del mismo punto y avanzan independiente.

Comandos de ramas

Crear, cambiar y listar ramas

Operaciones basicas

# Ver todas las ramas
git branch

# Crear una nueva rama
git branch feature/login

# Cambiar a esa rama
git switch feature/login

# Crear Y cambiar en un paso
git switch -c feature/login

# Volver a main
git switch main

# Borrar una rama (ya mergeada)
git branch -d feature/login

Nomenclatura de ramas

TipoNombre
Nueva funcionalidadfeature/nombre
Correccion de bugfix/descripcion
Urgente en produccionhotfix/descripcion
Releaserelease/1.2.0

HEAD — donde estas parado

HEAD es un puntero que siempre apunta al commit actual. Cuando cambias de rama, HEAD se mueve. git log te muestra (HEAD → main) para decirte donde estas.

Merge

Unir ramas con git merge

Merge — traer cambios de otra rama

# Quiero traer feature/login a main

# 1. Me paro en la rama destino
git switch main

# 2. Hago el merge
git merge feature/login

# Si todo bien: fast-forward o merge commit
# Si hay conflictos: Git te avisa

Fast-forward vs Merge commit

Fast-forward: si main no cambio mientras trabajabas en la feature, Git simplemente mueve el puntero. Merge commit: si ambas ramas avanzaron, Git crea un nuevo commit que une ambas historias.

Conflictos de merge

Ocurren cuando dos ramas modificaron la misma linea. Git no sabe cual version elegir.

<<<<<<< HEAD (tu main)
print("Hola mundo")
=======
print("Hello world")
>>>>>>> feature/traduccion

Editas el archivo, dejas la version correcta, y luego:

git add app.py
git commit -m "Resuelve conflicto de merge"
Modulo 04

Git vs GitHub

No son lo mismo — entender la diferencia es clave

La diferencia

Git vs GitHub

Git — la herramienta local

  • Software que corre en tu computadora
  • Guarda el historial de cambios localmente
  • No necesita internet para funcionar
  • Creado por Linus Torvalds en 2005
  • Completamente gratuito y open source

Sin GitHub, Git sigue funcionando. Es independiente.

Analogia

Git es como Word — el programa que edita documentos. GitHub es como Google Drive — el lugar donde guardas y compartes esos documentos. Podes usar Word sin Drive.

GitHub — la plataforma en la nube

  • Sitio web para alojar repositorios Git
  • Permite colaborar con otros programadores
  • Agrega features encima de Git: Issues, Pull Requests, Actions
  • Portfolio publico de tu codigo
  • Propiedad de Microsoft desde 2018

Alternativas: GitLab, Bitbucket, Gitea — misma idea.

Lo que GitHub agrega

Feature¿Que es?
IssuesReportar bugs y tareas
Pull RequestsRevision de codigo
ActionsCI/CD automatico
PagesHosting de sitios estaticos
GitHub

Subir tu repo local a GitHub

Paso 1 — Crear el repo en GitHub

  1. Ir a github.com e iniciar sesion
  2. Clic en "New repository" (boton verde o el +)
  3. Ponerle nombre al repo
  4. Dejarlo publico o privado
  5. No agregar README si ya tenes archivos locales
  6. Clic en "Create repository"

¿Publico o privado?

Publico: cualquiera puede ver el codigo (ideal para portfolio). Privado: solo vos y quienes invites. Ambos son gratis en GitHub.

Paso 2 — Conectar y subir

GitHub te da estos comandos al crear el repo:

# Conectar tu repo local con GitHub
git remote add origin https://github.com/
  usuario/mi-repo.git

# Renombrar la rama a main (si hace falta)
git branch -M main

# Subir por primera vez
git push -u origin main

# El -u guarda la configuracion
# Proximas veces solo necesitas:
git push

¿Que es origin?

origin es simplemente el nombre que le damos al remoto. Podes cambiarlo, pero por convencion siempre se llama origin. Verificas con git remote -v.

GitHub — la interfaz

Lo que encontras en un repo de GitHub

Pestana¿Para que?

Issues — el tablero de tareas

Cada issue es una tarea, bug o idea. Podes asignarlos a personas, ponerles etiquetas (bug, feature, help wanted) y cerrarlos automaticamente con commits:

git commit -m "Corrige login. Closes #42"
# El issue #42 se cierra automatico

Proteger la rama main

En Settings → Branches → Branch protection rules podes:

  • Exigir que todos los PRs tengan al menos 1 aprobacion
  • Exigir que los tests pasen antes de mergear
  • Prohibir push directo a main — todo via PR

Equipos serios siempre protegen main.

Clone vs Fork

Clone: bajas el repo para trabajar (necesitas permisos para hacer push). Fork: copias el repo completo a TU cuenta de GitHub. Asi podes contribuir a proyectos donde no sos colaborador — trabajas en tu fork y mandas un PR al original.

Modulo 05

Sincronizar con el remoto

push · pull · fetch — mantener todo en sync

Remotos

push · pull · fetch

El flujo con GitHub

# Ver remotos configurados
git remote -v

# Subir tus commits al remoto
git push origin main

# Subir una rama nueva por primera vez
git push -u origin feature/login

# Bajar cambios y mergear automatico
git pull

# Bajar cambios SIN mergear (para revisar)
git fetch

¿Que es origin?

origin es el nombre por defecto del repositorio remoto (el de GitHub). Podes tener varios remotos con distintos nombres, pero casi siempre hay uno solo llamado origin.

pull = fetch + merge

git pull baja los cambios y los mergea en tu rama actual automaticamente. git fetch solo los baja — te deja revisar antes de mergear. En equipos, fetch es mas seguro.

Antes de hacer push

# Siempre baja primero para evitar
# conflictos en el remoto
git pull origin main
git push origin main
Modulo 05

Gitflow

El flujo de trabajo estandar para trabajar en equipo

Gitflow

¿Que es Gitflow?

Un modelo de ramas para equipos

Gitflow es un conjunto de reglas sobre como nombrar y usar las ramas. No es una herramienta, es un acuerdo de como trabajar juntos.

Define que tipo de rama existe para cada proposito, cuando crearlas, cuando mergearlas y hacia donde.

¿Por que no commitear directo en main?

  • Main debe estar siempre estable y deployable
  • Dos personas en main al mismo tiempo = conflictos constantes
  • Sin estructura, el historial queda ilegible
  • No hay forma de revertir una feature sin afectar otras

Las 5 ramas de Gitflow

RamaProposito
mainProduccion — siempre estable
developIntegracion — codigo listo para release
feature/xNueva funcionalidad
release/xPreparar una version
hotfix/xBug urgente en produccion

El flujo de una feature

  1. Partir de develop
  2. Crear feature/mi-feature
  3. Trabajar y commitear en la feature
  4. Merge a develop cuando este lista
  5. Borrar la rama de feature

El flujo de un release

  1. Partir de develop
  2. Crear release/1.2.0
  3. Solo bugfixes (no features nuevas)
  4. Merge a main y a develop
  5. Tag de version en main (v1.2.0)

El flujo de un hotfix

  1. Bug critico en produccion
  2. Partir de main (no develop)
  3. Crear hotfix/descripcion
  4. Merge a main y a develop
Gitflow en la practica

El flujo completo en comandos

feature completa
# 1. Actualizo develop
git switch develop
git pull origin develop

# 2. Creo mi rama
git switch -c feature/login

# 3. Trabajo y commiteo
git add .
git commit -m "Agrega pantalla de login"
git commit -m "Conecta login con backend"

# 4. Actualizo con develop por si cambio
git switch develop
git pull origin develop
git switch feature/login
git merge develop

# 5. Mergeo a develop
git switch develop
git merge feature/login

# 6. Subo y borro la feature
git push origin develop
git branch -d feature/login

Por que mergear develop en la feature (paso 4)

Si develop avanzo mientras trabajabas, traes sus cambios a tu rama primero. Resuelves los conflictos ahi, no en develop. Cuando hagas el merge final a develop, va limpio.

Gitflow simplificado

Muchos equipos usan una version simplificada: solo main + ramas de features. Sin develop, sin release. Se llama GitHub Flow. Es mas simple y funciona bien para proyectos mas chicos o con deploy continuo.

GitHub Flow en 3 pasos

  1. Branch desde main
  2. Pull Request cuando este listo
  3. Merge a main y deploy
Pull Requests

¿Que es un Pull Request?

Un PR = pedido de revision

Cuando terminas una feature, en vez de mergear directamente, abrís un Pull Request en GitHub. Es una solicitud formal para que alguien revise tu codigo antes de mergearlo.

  • Se ve exactamente que lineas cambiaron
  • El equipo comenta, sugiere cambios
  • Se pueden correr tests automaticos
  • Cuando todos aprueban, se mergea

Flujo con PR

  1. Subis tu rama: git push origin feature/login
  2. Vas a GitHub y abris un Pull Request
  3. Elegis: feature/login → develop
  4. Describes que hace el PR y por que
  5. Companeros revisan y comentan
  6. Vos respondés y actualizas si hace falta
  7. Alguien aprueba → se mergea

Por que los PRs son importantes

Sin PRs, los bugs llegan directo a main. Con PRs, el codigo pasa por al menos otro par de ojos. En equipos profesionales, nadie mergea su propio codigo sin revision.

Ejercicio 1
Enunciado

Tu primer repo con historial

Instrucciones

  1. Crea una carpeta mi-primer-repo e inicializa Git
  2. Crea un archivo .gitignore con los patrones tipicos de Python
  3. Crea main.py con una funcion saludar(nombre)
  4. Haz tu primer commit: "Agrega funcion saludar"
  5. Agrega una funcion despedir(nombre) y haz un segundo commit
  6. Crea un bug a proposito (divide por cero), commitea
  7. Deshaz el ultimo commit con git reset --soft HEAD~1
  8. Verifica el historial con git log --oneline

Comandos que vas a usar

git add .gitignore
git commit -m "..."
git status
git log --oneline

Solucion

Tu primer repo con historial

terminal

mkdir mi-primer-repo
cd mi-primer-repo

# .gitignore
echo "__pycache__/\nvenv/\n*.pyc\n.env" > .gitignore

# (escribis main.py con saludar())

# Segundo commit
# (agregas despedir() a main.py)
git commit -am "Agrega funcion despedir"

# El bug
# (escribis el bug en main.py)
git commit -am "Divide por cero bug"

# Deshacer sin perder los cambios
git reset --soft HEAD~1

# Verificar historial
git log --oneline

main.py

def saludar(nombre):
    print(f"Hola, {nombre}!")

def despedir(nombre):
    print(f"Adios, {nombre}!")

saludar("Eduardo")
despedir("Eduardo")

¿Que aprendemos?

  • --soft deshace el commit pero mantiene los archivos modificados
  • El staging area queda con los cambios listos
  • Podes volver a commitear con el mensaje correcto

Feature branches y merge

Fast-forward vs merge commit

Si develop no cambio mientras trabajabas en la feature, Git hace fast-forward: mueve el puntero sin crear un commit extra. Para forzar siempre un merge commit: git merge --no-ff feature/x. Muchos equipos lo prefieren para que la historia quede clara.
