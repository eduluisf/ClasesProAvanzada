Git ≠ GitHub — son dos cosas distintas

Git es la herramienta que corre en tu computadora y lleva el historial. GitHub es el sitio web donde guardás ese historial en la nube. Primero entendemos Git, después conectamos GitHub.

Diferencia clave

Git vs GitHub — antes de arrancar

Git — la herramienta local

  • Software que corre en tu compu
  • Guarda el historial localmente
  • No necesita internet
  • Creado por Linus Torvalds en 2005
  • Completamente gratuito y open source

Podés usar Git sin GitHub. Al revés no.

GitHub — la plataforma en la nube

  • Sitio web para alojar repos Git en la nube
  • Permite colaborar con otras personas
  • Agrega features: Issues, Pull Requests, Actions
  • Sirve como portfolio de tu código
  • Estándar de la industria — el 95% de proyectos vive acá

Alternativas: GitLab, Bitbucket — misma idea.

La carpeta .git/ es la base de datos de Git. Todo el historial vive ahí. No la toques nunca a mano.

¿Qué hizo git init?

Initialized empty ← Repo vacío — sin commits todavía. Es normal.
Git repository in ← La ruta completa donde creó el repo
calculadora/.git/ ← Esta carpeta oculta ES el repo — ahí guarda todo el historial

Una sola vez por proyecto. Después nunca más.

git init ← solo al principio

¿Por qué NO agregar el README?

Porque ya tenemos un repo local con commits. Si GitHub crea un commit propio (el del README), los historiales chocan y no podés hacer push fácil.

Cuando arrancás desde GitHub y todavía no tenés nada local → tildar README. Cuando ya tenés repo local → dejarlo vacío.

Después de crear verás esto en GitHub

…or push an existing repository:

git remote add origin
  https://github.com/TU_USUARIO/calculadora.git
git branch -M main
git push -u origin main

GitHub mismo te dice los comandos. Vamos uno por uno.

Paso 6 — entender cada parte

git remote add — conectar los repos

terminal
git remote add origin https://github.com/TU_USUARIO/calculadora.git

Verificar que quedó

$ git remote -v
origin  https://github.com/TU_USUARIO/calculadora.git (fetch)
origin  https://github.com/TU_USUARIO/calculadora.git (push)

Dos líneas: una para bajar (fetch) y otra para subir (push). Ambas apuntan a la misma URL.

Palabra por palabra

git remote ← Gestionar las "conexiones remotas" del repo
add ← Agregar una nueva conexión
origin ← El alias que le damos a esa URL — podría llamarse cualquier cosa, pero "origin" es la convención universal
https://github.com/... ← La URL real del repo que creaste en GitHub

Una sola vez por repo. Después git push sabe a dónde ir porque ya tiene el alias.

Siguiente nivel

Ramas (Branches)

Trabajar en una feature sin tocar lo que ya funciona

El problema sin ramas

¿Por qué no trabajar directo en main?

Sin ramas — el problema

Querés agregar una feature nueva. Mientras la desarrollás está a la mitad y no funciona bien. Pero tu jefe necesita que la versión actual funcione y esté deployada ahora.

Si trabajás directo en main, tenés código roto en producción.

Con ramas — la solución

Creás una rama feature/validaciones. Trabajás ahí. main queda intacto y funcionando. Cuando la feature esté lista y probada, la unís a main.

Cómo se ve en el historial

main:         A ── B ────────── E
                         \       /
feature/x:                C ── D
A, B ← Commits en main que ya existían
C, D ← Commits en la feature — main no los tiene todavía
E ← El merge — une todo. Ahora main tiene C y D también.
Paso 8

Crear la rama y trabajar en ella

terminal
# Ver en qué rama estás
git branch
* main

# Crear la rama Y moverse en un paso
git switch -c feature/validaciones

# Verificar
git branch
  main
* feature/validaciones

El * dice dónde estás parado

Ahora tus commits van a esta rama. main queda exactamente como estaba.

calculadora.py — agregamos validación
def validar_numeros(a, b):
    if not isinstance(a, (int, float)):
        raise TypeError(f"{a} no es un número")
    if not isinstance(b, (int, float)):
        raise TypeError(f"{b} no es un número")

def sumar(a, b):
    validar_numeros(a, b)
    return a + b
commiteamos en la feature
git add calculadora.py
git commit -m "Agrega validacion de tipos"
Paso 9

Merge — unir la feature a main

terminal
# 1. Volver a main
git switch main

# 2. Traer los cambios de la feature
git merge feature/validaciones
output
Updating c44b1f3..d9e2a01
Fast-forward
 calculadora.py | 8 ++++++++
 1 file changed, 8 insertions(+)
limpiar y subir
git branch -d feature/validaciones
git push

Fast-forward — línea por línea

Updating c44..d9e ← main pasa del commit viejo al nuevo
Fast-forward ← main no había cambiado → Git simplemente mueve el puntero. No crea un commit extra.
8 insertions(+) ← Las 8 líneas nuevas que vinieron de la feature

git branch -d — limpiar

Una vez mergeada, la rama ya no es necesaria. La borrás localmente. En GitHub también podés borrarla desde la interfaz (te lo sugiere después del merge).

Trabajo en equipo

Gitflow — cómo lo hacen los equipos

Una convención de ramas para no pisarse entre todos

Gitflow

Cada rama tiene un rol específico

Las ramas y sus roles

RamaRol
mainProducción — siempre estable y deployable
developIntegración — acumula features para el próximo release
feature/xUna funcionalidad nueva (sale de develop)
release/xPreparar una versión (sale de develop)
hotfix/xBug urgente en producción (sale de main)

Flujo visual

main:      A ─────────────────── F ── H
                \                 ↑     ↑
develop:    B ── C ──── E ───── G     |
                  ↑       ↑           |
feature:    D ──/   D2 ──/     hotfix ─/
A, F, H — main ← Solo recibe releases y hotfixes — nunca features directas
B...G — develop ← Recibe todas las features — la "cocina" del equipo

Regla de oro de Gitflow

Nadie commitea directo en main. Todo entra via Pull Request. main solo recibe merges desde release o hotfix.

GitHub — la killer feature

Pull Requests — revisión de código

¿Qué es un PR?

En vez de mergear directo, abrís una solicitud formal en GitHub para que alguien revise tu código antes de que entre a develop o main.

1

git push origin feature/mi-feature

2

GitHub te muestra "Compare & pull request" — lo abrís

3

Escribís título y descripción: qué hace y por qué

4

Un compañero revisa, comenta, pide cambios o aprueba

5

Se mergea — aparece en develop o main

Por qué los PRs son importantes

  • Nadie mergea su propio código sin revisión
  • Los bugs se atajan antes de llegar a main
  • Todo el equipo sabe qué está entrando y por qué
  • Queda registro de las decisiones de diseño

Fork vs Clone

Clone: bajás el repo para trabajar (necesitás permisos para hacer push). Fork: copiás el repo completo a TU cuenta de GitHub. Así podés contribuir a proyectos donde no sos colaborador — trabajás en tu fork y mandás un PR al original.

Cuando algo sale mal

Deshacer sin entrar en pánico

Situación 1 — modifiqué un archivo, quiero volver

# Archivo modificado, lo quiero como estaba
git restore calculadora.py

# Quitar del staging (sin perder los cambios)
git restore --staged calculadora.py

Situación 2 — commiteé algo mal

# Deshace el commit — mantiene los cambios
# en staging para que puedas recomitear
git reset --soft HEAD~1

# HEAD~1 = "el commit anterior al actual"

Situación 3 — ya hice push, necesito deshacer

# NO uses reset después del push
# Usá revert: crea un commit que deshace
git revert 7b2e441

# Crea un nuevo commit que invierte los cambios
# del commit 7b2e441 — sin borrar la historia
reset ← Borra commits de la historia — peligroso si ya hiciste push
revert ← Agrega un commit que deshace — seguro siempre

tareas.py — completo
tareas = []

def agregar(tarea):
    tareas.append(tarea)
    print(f"✓ Agregada: {tarea}")

def listar():
    if not tareas:
        print("No hay tareas.")
        return
    for i, t in enumerate(tareas, 1):
        print(f"{i}. {t}")

def eliminar(indice):
    try:
        t = tareas.pop(indice - 1)
        print(f"Eliminada: {t}")
    except IndexError:
        print("Índice inválido")
