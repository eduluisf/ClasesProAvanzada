¿Por que existe Git?

¿Reconoces alguno de estos archivos?

La carpeta de cualquiera sin Git

📁 proyecto/
   main.py
   main_v2.py
   main_v2_BUENO.py
   main_ESTE_SI.py
   main_ESTE_SI_final.py
   main_final_DEFINITIVO.py
   main_final_DEFINITIVO2.py

¿Cual era el bueno? ¿Que cambio entre versiones? ¿Podes volver a una version anterior sin perder nada?

Con Git, solo necesitas uno

📁 proyecto/
   main.py       ← el unico archivo

📋 Historial Git:
   a3f9c — Agrega login     ← hoy
   7b2e4 — Arregla bug      ← ayer
   c91d0 — Agrega registro  ← la semana pasada
   1a4f8 — Primer commit    ← hace un mes

Podes volver a cualquier punto. Cada cambio tiene fecha, autor y descripcion. Eso es Git.

Git ≠ GitHub

Git corre en tu compu, guarda la historia localmente. GitHub es el sitio web donde subis esa historia a la nube para compartirla. Hoy usamos los dos.

Paso 1

Crear la carpeta e inicializar Git

terminal
# Crear la carpeta del proyecto
mkdir calculadora
cd calculadora

# Inicializar Git aqui
git init

Lo que Git responde

Initialized empty Git repository in
  /Users/vos/calculadora/.git/

Git creo una carpeta oculta .git/ adentro. Ahi vive toda la historia. Nunca la toques a mano.

¿Que hay en .git/?

📁 calculadora/
   📁 .git/          ← Git vive aca
      HEAD
      config
      objects/       ← los snapshots
      refs/          ← los punteros de ramas
   (por ahora solo esto)

¿Que significa "empty repository"?

Git esta inicializado pero todavia no hay ningun commit. No hay historia aun. El repo existe pero esta vacio — como un cuaderno en blanco.

Paso 2

Crear el primer archivo

calculadora.py
def sumar(a, b):
    return a + b

def restar(a, b):
    return a - b

print(sumar(5, 3))   # 8
print(restar(10, 4)) # 6

Crea este archivo en tu carpeta calculadora/.

Ahora corri git status

$ git status

On branch main

No commits yet

Untracked files:
  (use "git add ..." to include)
        calculadora.py

nothing added to commit but untracked
files present

¿Que dice?

  • Untracked — Git ve el archivo pero no lo esta rastreando todavia
  • Git no guarda nada automaticamente
  • Tenes que decirle explicitamente que queres guardar
Paso 3

git add — preparar el archivo

terminal
git add calculadora.py

Ahora git status dice algo diferente

$ git status

On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached ..." to unstage)
        new file:   calculadora.py

El archivo paso de rojo a verde. Ahora esta en el staging area — listo para ser parte del proximo commit.

Las 3 zonas de Git

● Working Dir — editas archivos aca
git add ↓
● Staging Area — preparas los cambios
git commit ↓
● Repositorio — historia guardada

¿Por que existe el staging?

Podes tener 10 archivos modificados y commitear solo 3. El staging te deja elegir exactamente que entra en cada commit. Un commit = un cambio logico, no todos tus cambios del dia.

Paso 4

git commit — guardar la foto

terminal
git commit -m "Agrega funciones sumar y restar"

Git responde

[main (root-commit) a3f9c12]
  Agrega funciones sumar y restar
 1 file changed, 7 insertions(+)

Git guardo un snapshot completo de todos los archivos en este momento. Ese a3f9c12 es el ID unico del commit — como una huella digital.

Buenos mensajes de commit

MalBien
cambiosAgrega funcion sumar
aaaCorrige bug en restar
fixRefactoriza validaciones
WIPAgrega multiplicar y dividir

El mensaje es una nota para vos del futuro (y para tu equipo). Usa el imperativo: "Agrega", "Corrige", "Elimina".

Viste lo que paso

El commit genero un ID (a3f9c12), guardo quien lo hizo, cuando y que cambio. Ahora esa foto existe para siempre en la historia local.

Paso 5

git log — ver la historia

terminal
git log --oneline

Git muestra

a3f9c12 (HEAD → main) Agrega funciones sumar y restar

Un solo commit por ahora. HEAD → main dice que estas parado en la rama main, en el commit mas reciente.

Variantes utiles de git log

# Compact — el mas usado
git log --oneline

# Ver que cambio en cada commit
git log -p

# Con grafico de ramas
git log --oneline --graph --all

# Solo commits de una persona
git log --author="Eduardo"

¿Y si me equivoque en el commit?

# Deshace el commit pero
# mantiene los archivos modificados
git reset --soft HEAD~1

# HEAD~1 = "el commit anterior al actual"

Es como "ctrl+z" del commit. Los archivos quedan igual, solo deshace el snapshot.

Paso 8

git remote y git push

terminal — en tu carpeta calculadora/
# Conectar tu repo local con GitHub
git remote add origin https://github.com/
  TU_USUARIO/calculadora.git

# Verificar que quedo bien
git remote -v

# Subir todo a GitHub
git push -u origin main

Git responde

Enumerating objects: 6, done.
Counting objects: 100%...
Writing objects: 100%...
Branch 'main' set up to track 'origin/main'

¿Que es origin?

origin es el nombre que le damos al repositorio remoto. Podes verlo como un "apodo" para la URL de GitHub. Por convencion siempre se llama origin.

$ git remote -v
origin  https://github.com/usuario/calculadora.git (fetch)
origin  https://github.com/usuario/calculadora.git (push)

El -u de git push -u

El -u configura el "upstream" — le dice a Git que de ahora en mas, cuando hagas git push a secas, sepa que va a origin main. Solo lo haces la primera vez.

# Primera vez:
git push -u origin main

# Todas las siguientes veces:
git push

Siguiente nivel

Ramas (Branches)

Experimentar sin romper lo que funciona

Branches

¿Por que necesitamos ramas?

El problema sin ramas

Queres agregar una feature nueva a tu calculadora. Mientras la desarrollas, esta a la mitad y no funciona bien. Pero tu jefe necesita que la version actual siga andando bien y deployada.

Si trabajas directo en main, tenes un problema.

La solucion: una rama separada

Creas una rama feature/historial. Trabajas ahi. main sigue intacto y funcionando. Cuando la feature este lista y probada, la unes a main.

Como se ve

main:
  A ── B ── C ──────────── F
                 \         /
feature/x:        D ── E

A,B,C = commits en main
D,E = commits en feature/x
F = merge (une todo)

Una rama = una linea de desarrollo

Cada rama tiene su propio historial. Pueden avanzar en paralelo. Cuando unes dos ramas es un merge.

Paso 9

Crear una rama y trabajar en ella

terminal
# Ver en que rama estas
git branch
* main

# Crear la rama Y moverse a ella
git switch -c feature/historial

# Verificar
git branch
  main
* feature/historial

El asterisco (*) dice donde estas

Ahora tus commits van a esta rama, no a main. Main esta intacto.

calculadora.py — le agregamos historial
def sumar(a, b):
    return a + b

def restar(a, b):
    return a - b

historial = []

def calcular(operacion, a, b):
    ops = {
        "sumar": sumar,
        "restar": restar,
    }
    resultado = ops[operacion](a, b)
    historial.append(f"{operacion}({a},{b})={resultado}")
    return resultado
commiteamos en la feature
git add calculadora.py
git commit -m "Agrega historial de operaciones"
Paso 10

Merge — unir la rama a main

terminal
# 1. Volver a main
git switch main

# 2. Unir la feature a main
git merge feature/historial

# Git responde:
Updating 7b2e441..c44b1f3
Fast-forward
 calculadora.py | 12 ++++++++++
 1 file changed, 12 insertions(+)

# 3. Borrar la rama (ya no la necesitamos)
git branch -d feature/historial

# 4. Subir el merge a GitHub
git push

¿Que es "Fast-forward"?

main no habia cambiado mientras trabajabas en la feature. Git simplemente mueve el puntero de main al ultimo commit de la feature. Es el caso mas limpio.

Antes:  main → A ── B
                         \
               feature → C ── D

Despues: main ────────────── D  ← fast-forward

Ver el historial final

$ git log --oneline
c44b1f3 (HEAD → main, origin/main) Agrega historial
7b2e441 Agrega multiplicar y dividir
a3f9c12 Agrega funciones sumar y restar
Cuando hay conflicto

¿Que pasa si dos ramas tocaron la misma linea?

Git no puede decidir por vos

Si main y una feature cambiaron la misma linea, Git te avisa y te deja resolver el conflicto:

<<<<<<< HEAD (main)
    return a + b
======
    result = a + b
    return result
>>>>>>> feature/refactor

Editas el archivo y dejas la version correcta. Despues:

git add calculadora.py
git commit -m "Resuelve conflicto en sumar"

Como evitar conflictos

  • Commits chicos y frecuentes
  • Actualiza tu rama con main antes de hacer merge: git merge main dentro de tu feature
  • Coordina con el equipo quien toca que archivos
  • Features chicas = menos conflictos

Los conflictos son normales

Todos los equipos tienen conflictos. Git los marca claramente. No es una falla — es el sistema avisandote que dos personas tocaron lo mismo. El conflicto nunca se pierde trabajo, siempre podras ver ambas versiones.

Trabajar en equipo

Gitflow — el flujo organizado

Como estructuran las ramas los equipos profesionales

Gitflow

Las ramas con un rol especifico

El problema sin estructura

Si 5 personas trabajan commiteando directo en main, hay conflictos todo el tiempo, el historial es un desastre y nunca sabes si main esta estable o roto.

Gitflow define 5 tipos de ramas

RamaRol
mainProduccion — siempre estable
developIntegracion — el siguiente release
feature/xUna funcionalidad nueva
release/xPreparar una version
hotfix/xBug urgente en produccion

La regla principal

Nadie commitea directo en main. Todo entra via Pull Request desde otra rama. Main solo recibe merges de release o hotfix.

El flujo de una feature

develop → feature/login → develop → release/1.0 → main
  1. Abris feature/login desde develop
  2. Trabajas y commiteas ahi
  3. Cuando lista, merge a develop
  4. Cuando se acumula suficiente, release a main
Gitflow — comandos

El flujo completo en la terminal

feature completa
# 1. Siempre arrancar actualizado
git switch develop
git pull origin develop

# 2. Nueva rama desde develop
git switch -c feature/nueva-operacion

# 3. Trabajar y commitear
git add .
git commit -m "Agrega modulo"
git commit -m "Agrega tests"

# 4. Traer develop por si cambio
git switch develop && git pull
git switch feature/nueva-operacion
git merge develop  # traer lo nuevo

# 5. Merge a develop
git switch develop
git merge feature/nueva-operacion

# 6. Subir y limpiar
git push origin develop
git branch -d feature/nueva-operacion

El hotfix — urgente en produccion

# Bug en produccion — parten de MAIN
git switch main
git switch -c hotfix/bug-dividir

# Corrigen el bug
git commit -am "Corrige division por cero"

# Merge a main Y a develop
git switch main
git merge hotfix/bug-dividir
git switch develop
git merge hotfix/bug-dividir

git branch -d hotfix/bug-dividir

GitHub Flow — version simplificada

Proyectos mas chicos usan GitHub Flow: solo main + ramas de feature. Sin develop ni release. Feature lista → Pull Request → merge a main → deploy. Simple y efectivo.

GitHub — la estrella

Pull Requests — revision de codigo

¿Que es un PR?

En vez de mergear directo, abris una solicitud formal en GitHub para que alguien revise tu codigo antes de que entre a develop o main.

1

git push origin feature/mi-feature

2

Vas a GitHub → tu repo → "Compare & pull request"

3

Titulo y descripcion: que hace y por que

4

Companero revisa, comenta, aprueba

5

Se mergea — aparece en develop

github.com / usuario / calculadora / pull / 3

Agrega historial de operaciones
Open feature/historial → develop
Files changed: 1  ·  +12 −0
+ historial = []
+ def calcular(operacion, a, b):
✅ 1 review requested  ·  🟡 Checks passing

Por que son importantes

  • Nadie mergea su propio codigo sin revision
  • Los bugs se atajan antes de llegar a main
  • El equipo esta al tanto de los cambios

tareas.py — funciones nuevas
completadas = set()

def completar_tarea(indice):
    completadas.add(indice - 1)
    print(f"Tarea {indice} completada.")

def eliminar_tarea(indice):
    try:
        eliminada = tareas.pop(indice - 1)
        print(f"Eliminada: {eliminada}")
    except IndexError:
        print("Indice invalido")
