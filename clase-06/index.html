Repaso

¿Donde quedamos?

Threading resuelve I/O bound — pero hay un limite

El GIL — el candado de Python

¿Por que threading no escala para CPU?

El GIL (Global Interpreter Lock)

Python tiene un candado global que permite que solo un hilo ejecute bytecode Python a la vez, aunque haya 8 nucleos disponibles.

  • Protege la integridad interna del interprete
  • Es transparente para codigo I/O (la espera libera el GIL)
  • Para calculos puros, los hilos no ayudan — compiten por el mismo candado

Demostracion: mas hilos, mismo tiempo

# tarea CPU-intensiva
def calcular(n):
    return sum(i*i for i in range(n))

# Con threads: NO mejora
# Todos compiten por el GIL
# 4 threads ≈ 1 thread en CPU-bound

La solucion: procesos separados

Cada proceso tiene su propio interprete Python y su propio GIL. No compiten entre si. Pueden usar nucleos diferentes realmente al mismo tiempo.

Regla de oro

¿Tu programa espera? (red, archivo, DB) → threading. ¿Tu programa calcula? (numeros, imagenes, ML) → multiprocessing. El costo de arrancar un proceso solo vale la pena si el trabajo tarda lo suficiente.

Modulo 02

multiprocessing — el modulo

Crear procesos separados que usan nucleos reales

Basico

Crear procesos con multiprocessing

proceso_simple.py
import multiprocessing
import time

def calcular(nombre, n):
    print(f"{nombre}: calculando...")
    resultado = sum(i*i for i in range(n))
    print(f"{nombre}: listo! = {resultado}")

if __name__ == "__main__":
    p1 = multiprocessing.Process(
        target=calcular,
        args=("A", 5_000_000)
    )
    p2 = multiprocessing.Process(
        target=calcular,
        args=("B", 5_000_000)
    )

    p1.start()
    p2.start()

    p1.join()   # esperar a que termine
    p2.join()

Partes clave

  • Process(target=fn, args=(...)) — define el proceso
  • .start() — lanza el proceso (no bloquea)
  • .join() — espera a que ese proceso termine
  • Cada proceso corre en un nucleo diferente de verdad

if __name__ == "__main__" — OBLIGATORIO

En Windows y Mac, el modulo multiprocessing importa tu script al crear nuevos procesos. Sin esta guarda, el codigo se ejecutaria recursivamente infinitas veces. Siempre ponlo.

Resultado

B: calculando...
# Ambos a la vez en nucleos distintos
B: listo! = ...
A: listo! = ...

La diferencia con threading

Con threads para CPU-bound, los 4 hilos compiten por un nucleo (GIL). Con procesos, cada uno tiene su nucleo. Son literalmente simultaneos.

Modulo 03

ProcessPoolExecutor

La misma API que ThreadPoolExecutor — pero con procesos

API de alto nivel

ProcessPoolExecutor — la forma recomendada

pool_procesos.py
from concurrent.futures import (
    ProcessPoolExecutor, as_completed
)
import time

def trabajo_pesado(n):
    # Simula calculo intensivo
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    numeros = [5_000_000] * 4

    t = time.time()
    with ProcessPoolExecutor() as pool:
        resultados = list(
            pool.map(trabajo_pesado, numeros)
        )
    total = time.time() - t

    print(f"Resultados: {resultados}")
    print(f"Tiempo: {total:.2f}s")

Identico a ThreadPoolExecutor

Solo cambias Thread por Process en la clase. Misma API: map(), submit(), as_completed(), bloque with.

max_workers por defecto

Si no especificas max_workers, usa todos los nucleos de tu CPU. Puedes verlo con:

import os
print(os.cpu_count())  # ej: 8

Ojo: datos serializables

Los datos viajan entre procesos via pickle. Lo que le pases a la funcion (y lo que retorne) debe poder serializarse: numeros, strings, listas, dicts. Lambdas y closures no funcionan.

submit + as_completed

Mas control con submit()

submit_procesos.py
from concurrent.futures import (
    ProcessPoolExecutor, as_completed
)

def calcular_primo(n):
    """Cuenta primos hasta n (CPU-intensivo)"""
    def es_primo(x):
        if x < 2: return False
        for i in range(2, int(x**0.5)+1):
            if x % i == 0: return False
        return True
    return n, sum(1 for x in range(2, n) if es_primo(x))

if __name__ == "__main__":
    limites = [100_000, 200_000, 300_000, 400_000]
    with ProcessPoolExecutor() as pool:
        futuros = {pool.submit(calcular_primo, n): n
                   for n in limites}
        for f in as_completed(futuros):
            n, cantidad = f.result()
            print(f"Hasta {n}: {cantidad} primos")

Mismo patron que con threads

  • submit(fn, arg) lanza el calculo en un proceso
  • as_completed() itera segun terminan
  • El resultado viaja de vuelta via pickle
  • El proceso hijo desaparece al terminar

Resultado esperado

Hasta 100000: 9592 primos
Hasta 200000: 17984 primos
Hasta 300000: 25445 primos
Hasta 400000: 33860 primos
# Orden puede variar (as_completed)

¿Por que no lambda?

Las lambdas no son picklables. La funcion pasada a submit() debe ser definida en el nivel top del modulo, no dentro de otra funcion.

Benchmark real

Threads vs Procesos — midiendo la diferencia

benchmark.py
import time
from concurrent.futures import (
    ThreadPoolExecutor,
    ProcessPoolExecutor
)

def cpu_trabajo(n):
    return sum(i * i for i in range(n))

N = [3_000_000] * 4

if __name__ == "__main__":
    # Secuencial
    t1 = time.time()
    [cpu_trabajo(n) for n in N]
    t_sec = time.time() - t1

    # Con threads (GIL limita)
    t2 = time.time()
    with ThreadPoolExecutor(4) as p:
        list(p.map(cpu_trabajo, N))
    t_thr = time.time() - t2

    # Con procesos (paralelo real)
    t3 = time.time()
    with ProcessPoolExecutor(4) as p:
        list(p.map(cpu_trabajo, N))
    t_proc = time.time() - t3

    print(f"Secuencial: {t_sec:.2f}s")
    print(f"Threads:    {t_thr:.2f}s")
    print(f"Procesos:   {t_proc:.2f}s")

Resultado tipico (4 nucleos)

Secuencial: 4.20s
Threads:    4.18s  ← igual
Procesos:   1.12s  ← 3.7x mas rapido

La conclusion es clara

  • Threads no ayudan en CPU-bound — el GIL los bloquea
  • Procesos usan todos los nucleos — mejora real proporcional
  • La mejora es cercana a N × velocidad_1_nucleo / N

¿Por que no exactamente 4x?

Hay overhead al crear procesos, serializar datos y comunicar resultados. Para tareas muy cortas, ese costo puede superar el beneficio.

Modulo 04

Comunicacion entre procesos

Los procesos no comparten memoria — hay que pasarse mensajes

Memoria aislada

¿Por que los procesos no comparten variables?

La trampa clasica

Esto no funciona como esperas:

from multiprocessing import Process

contador = 0

def incrementar():
    global contador
    contador += 1  # modifica UNA COPIA

if __name__ == "__main__":
    p = Process(target=incrementar)
    p.start()
    p.join()
    print(contador)  # imprime 0, no 1

¿Por que?

Al crear el proceso hijo, Python copia toda la memoria del padre. Cada proceso trabaja con su propia copia. Los cambios en el hijo no se ven en el padre.

Memoria de procesos — aislada

Proceso padre:
┌─────────────────────┐
│  contador = 0       │ ← no cambia
└─────────────────────┘
         |
         | fork (copia)
         ↓
Proceso hijo:
┌─────────────────────┐
│  contador = 0       │
│  contador = 1       │ ← su copia local
└─────────────────────┘
         |
         | (proceso termina, copia desaparece)

La solucion

Para compartir datos entre procesos se usan mecanismos explícitos: Queue (cola de mensajes), Pipe (canal bidireccional) o Value/Array (memoria compartida con sincronizacion).

Queue

Comunicar procesos con Queue

queue_ejemplo.py
from multiprocessing import Process, Queue

def calcular_y_enviar(n, cola):
    resultado = sum(i*i for i in range(n))
    cola.put((n, resultado))  # envia el resultado

if __name__ == "__main__":
    cola = Queue()

    procesos = [
        Process(target=calcular_y_enviar,
                args=(n, cola))
        for n in [1_000_000, 2_000_000, 3_000_000]
    ]

    for p in procesos: p.start()
    for p in procesos: p.join()

    # Recoger resultados de la cola
    while not cola.empty():
        n, res = cola.get()
        print(f"n={n}: {res}")

¿Como funciona la Queue?

  • Es como un tubo entre procesos
  • cola.put(valor) — el hijo manda un dato
  • cola.get() — el padre lo recibe
  • Es thread-safe y process-safe
  • Los datos se serializan automaticamente

Alternativa simple: ProcessPoolExecutor

Con ProcessPoolExecutor.map() no necesitas Queue — los resultados se recogen automaticamente. Usa Queue solo cuando necesitas comunicacion mas compleja entre procesos.

Recomendacion practica

Para la mayoria de casos usa ProcessPoolExecutor. Recurre a Queue solo si los procesos necesitan comunicarse durante su ejecucion, no solo al final.

Modulo 05 — Bonus

asyncio — la tercera via

Concurrencia cooperativa sin hilos ni procesos

asyncio

Concurrencia cooperativa — un solo hilo

La idea

Con asyncio hay un bucle de eventos en un solo hilo. Las tareas se turnan voluntariamente: cuando una dice "voy a esperar", el bucle avanza con otra.

  • Sin GIL porque es un solo hilo
  • Sin overhead de crear procesos o hilos
  • Pero requiere que todo el codigo sea async-aware

Cuándo usar asyncio

  • I/O bound con muchisimas conexiones simultaneas (miles)
  • Servidores web de alto rendimiento
  • Cuando las librerias que usas ya son async (httpx, aiohttp)
asyncio_ejemplo.py
import asyncio
import httpx  # pip install httpx

async def descargar(url):
    async with httpx.AsyncClient() as c:
        resp = await c.get(url)
        return resp.json()["title"]

async def main():
    base = "https://jsonplaceholder.typicode.com"
    urls = [f"{base}/posts/{i}"
            for i in range(1, 6)]

    tareas = [descargar(u) for u in urls]
    resultados = await asyncio.gather(*tareas)

    for r in resultados:
        print(r)

asyncio.run(main())
asyncio — vocabulario

Las palabras clave de asyncio

async def

Define una corrutina — una funcion que puede pausarse y reanudarse. No se ejecuta al llamarla; devuelve un objeto corrutina.

async def mi_func():
    print("hola")

# Esto NO imprime nada:
mi_func()
# Necesitas await o run()

await

Pausa la corrutina actual y le cede el control al bucle de eventos hasta que la operacion termine.

async def tarea():
    # Pausa y espera
    await asyncio.sleep(1)
    # Reanuda despues
    print("listo")

asyncio.gather()

Lanza multiples corrutinas al mismo tiempo y espera a que todas terminen. Es el equivalente de pool.map().

await asyncio.gather(
    tarea1(),
    tarea2(),
    tarea3(),
) # las tres a la vez

La gran diferencia

Con threads el SO decide cuando cambiar de tarea (preemptivo). Con asyncio tu codigo decide cuando ceder con await (cooperativo). Si una corrutina no hace await, bloquea todo.

Los tres modelos

Threading · Multiprocessing · asyncio

threading

  • I/O bound
  • Facil de usar
  • Memoria compartida
  • GIL limita CPU-bound
  • Libreria: requests

Usa cuando: pocas tareas I/O, codigo existente

multiprocessing

  • CPU bound
  • Paralelo real
  • Memoria aislada
  • Overhead de arranque
  • No lambdas

Usa cuando: calculos pesados, ML, imagenes

asyncio

  • I/O bound
  • Muy escalable
  • Un solo hilo
  • Requiere async/await
  • Libreria: httpx, aiohttp

Usa cuando: miles de conexiones, servidores

Para el dia a dia

ThreadPoolExecutor para APIs y I/O. ProcessPoolExecutor para calculos pesados. asyncio si construyes una aplicacion que necesita manejar muchas conexiones simultaneas (un servidor, un bot, etc.).

Modulo 06

Cuando NO usar paralelismo

Mas complejidad no siempre significa mas velocidad

El overhead

El costo de crear procesos

Crear un proceso no es gratis

Cada proceso nuevo requiere:

  • Inicializar un nuevo interprete Python
  • Copiar la memoria del proceso padre
  • Importar los modulos necesarios
  • Serializar los datos a enviar (pickle)

Esto toma entre 50ms y 200ms por proceso. Si tu tarea tarda 10ms, el overhead es mayor que el trabajo.

Regla practica

Duracion de la tarea¿Usar procesos?
< 100ms por tareaProbablemente no
100ms – 1s por tareaDepende — mide
> 1s por tareaSi, vale la pena

Mide siempre antes de optimizar

La optimizacion prematura es la raiz de muchos problemas. Primero haz que funcione, luego mide con time.time(), y solo entonces optimiza si el tiempo importa.

Errores comunes

Lo que falla tipicamente

1. Olvidar if __name__ == "__main__"

# MAL — crash en Windows/Mac
p = Process(target=mi_fn)
p.start()

# BIEN
if __name__ == "__main__":
    p = Process(target=mi_fn)
    p.start()

2. Pasar objetos no serializables

# MAL — lambda no es picklable
with ProcessPoolExecutor() as p:
    p.map(lambda x: x*2, datos)

# BIEN — funcion definida arriba
def duplicar(x): return x * 2
with ProcessPoolExecutor() as p:
    p.map(duplicar, datos)

3. Crear demasiados procesos

No hagas max_workers=100 para 100 tareas. Cada proceso usa RAM y compite por nucleos. El optimo suele ser igual a tu numero de nucleos (os.cpu_count()).

4. Usar multiprocessing para I/O

Multiprocessing para descargar 100 URLs: funciona, pero crea 100 procesos pesados cuando ThreadPoolExecutor con 10 threads es mucho mas eficiente. Herramienta equivocada = resultado peor.

Ejercicio 1
Solucion

Secuencial vs Paralelo

ejercicio1.py
from concurrent.futures import ProcessPoolExecutor
import time

def es_primo(n):
    if n < 2: return False
    for i in range(2, int(n**0.5)+1):
        if n % i == 0: return False
    return True

def contar_primos(limite):
    return sum(1 for n in range(2, limite)
               if es_primo(n))

limites = [500_000, 600_000, 700_000, 800_000]

if __name__ == "__main__":
    t1 = time.time()
    [contar_primos(n) for n in limites]
    sec = time.time() - t1

    t2 = time.time()
    with ProcessPoolExecutor() as pool:
        list(pool.map(contar_primos, limites))
    par = time.time() - t2

    print(f"Secuencial: {sec:.2f}s")
    print(f"Paralelo:   {par:.2f}s")
    print(f"Speedup: {sec/par:.1f}x")

Lo que aprendemos

  • El cambio es minimo: for → pool.map()
  • Las funciones auxiliares (es_primo) deben estar en el nivel top
  • El if __name__ es necesario

¿Por que no es 4x exacto?

  • Overhead de crear los 4 procesos
  • Pickle para serializar los argumentos
  • Coordinacion del pool
  • Competencia en la cache del CPU

Clave

El speedup real depende de cuantos nucleos tiene tu maquina. En una maquina con 8 nucleos, el beneficio puede ser mayor si el pool usa todos.

Ejercicio 2
Enunciado

Elegir la herramienta correcta

El escenario

Tienes dos tareas independientes que queres hacer "al mismo tiempo":

  1. Tarea A — Descargar 10 posts de la API de JSONPlaceholder (I/O bound)
  2. Tarea B — Contar primos hasta 300_000 (CPU bound)

El objetivo: hacer ambas tareas de forma concurrente/paralela y medir cuanto tiempo ahorramos vs hacerlas en secuencia.

Instrucciones

  1. Implementa la Tarea A con ThreadPoolExecutor
  2. Implementa la Tarea B con ProcessPoolExecutor
  3. Ejecuta ambas tareas una por una y mide el tiempo total
  4. Ahora ejecuta ambas tareas usando ThreadPoolExecutor (para ambas) y mide
  5. Compara los tiempos y razona: ¿fue la combinacion mejor?

Pista

Para lanzar la Tarea A y la Tarea B al mismo tiempo puedes usar ThreadPoolExecutor para ambas y correr cada una en un hilo separado — la Tarea B aun podria beneficiarse de liberar el GIL durante la espera I/O de A.

Ejercicio 2
Solucion

Combinando threading y multiprocessing

ejercicio2.py
import requests, time
from concurrent.futures import (
    ThreadPoolExecutor, ProcessPoolExecutor, as_completed
)

def descargar_post(i):
    base = "https://jsonplaceholder.typicode.com"
    return requests.get(f"{base}/posts/{i}").json()["title"]

def es_primo(n):
    if n < 2: return False
    for i in range(2, int(n**0.5)+1):
        if n % i == 0: return False
    return True

def contar_primos(limite):
    return sum(1 for n in range(2, limite) if es_primo(n))

def tarea_io():
    ids = range(1, 11)
    with ThreadPoolExecutor(10) as p:
        return list(p.map(descargar_post, ids))

def tarea_cpu():
    with ProcessPoolExecutor() as p:
        return list(p.map(contar_primos,
                         [300_000]*4))

if __name__ == "__main__":
    t1 = time.time()
    tarea_io()
    tarea_cpu()
    sec = time.time() - t1

    t2 = time.time()
    with ThreadPoolExecutor(2) as outer:
        f1 = outer.submit(tarea_io)
        f2 = outer.submit(tarea_cpu)
        f1.result(); f2.result()
    con = time.time() - t2

    print(f"Secuencial: {sec:.2f}s")
    print(f"Concurrente: {con:.2f}s")

La idea clave

  • Usamos un ThreadPoolExecutor externo solo para lanzar ambas tareas al mismo tiempo
  • Dentro de tarea_io(), el threading real maneja la red
  • Dentro de tarea_cpu(), multiprocessing maneja el CPU
  • Ambas tareas corren solapadas en el tiempo

Lo que esto demuestra

Puedes mezclar modelos. Un hilo puede lanzar procesos. Un proceso puede lanzar threads. El limite es no crear demasiados de cada uno y entender que herramienta resuelve que problema.

Ejercicio 3 — Bonus
Solucion

asyncio con httpx

ejercicio3.py
import asyncio
import httpx

BASE = "https://jsonplaceholder.typicode.com"

async def descargar(client, url):
    resp = await client.get(url)
    return resp.json()["title"]

async def main():
    urls = [f"{BASE}/posts/{i}"
            for i in range(1, 11)]

    # Un solo cliente, multiples requests
    async with httpx.AsyncClient() as client:
        tareas = [descargar(client, u) for u in urls]
        titulos = await asyncio.gather(*tareas)

    for titulo in titulos:
        print(titulo)

asyncio.run(main())

¿Que hace cada parte?

  • async with httpx.AsyncClient() — cliente reutilizable que no bloquea
  • await client.get(url) — pausa la corrutina, el loop sigue con otra
  • asyncio.gather(*tareas) — lanza todas las corrutinas a la vez
  • El cliente se comparte entre todas las tareas — mas eficiente

threading vs asyncio para I/O

Para 10 requests, ambos son similares en velocidad. Para miles de conexiones simultaneas, asyncio gana: un hilo puede manejar miles de corrutinas, mientras que crear miles de threads es costoso en memoria.

Resumen

Lo que vimos en esta clase

El GIL

  • Solo un hilo ejecuta Python a la vez

ProcessPoolExecutor

  • Misma API que ThreadPoolExecutor

  • Un hilo, muchas corrutinas
